# 通用编码规范

本文件定义所有项目默认共享的编码约束。

它不是语言手册，而是默认下限，主要回答：
- 代码应该怎么组织
- 哪些实现习惯默认推荐
- 提交前至少应自检什么

## 基本原则

1. 代码先求可读，再求花哨。
2. 一个文件和一个函数都应尽量聚焦单一职责。
3. 默认在系统边界做输入校验。
4. 错误不能静默吞掉。
5. 尽量避免共享可变状态。

## 不可变性

优先创建新对象，而不是原地修改共享对象。

```text
错误：modify(original, field, value)
正确：update(original, field, value) -> new copy
```

为什么这样做：
- 降低隐藏副作用
- 减少调试成本
- 更适合并发和跨层传递

## 文件组织

默认建议：
- 高内聚、低耦合
- 按功能或领域组织，而不是按技术类型平铺
- 多个小文件通常优于少量超大文件

经验性参考：
- 单文件 200-400 行更容易维护
- 超过 800 行通常值得拆分

## 错误处理

默认要求：
- 每个层级都明确处理错误
- 用户侧返回可理解信息
- 服务端保留足够诊断信息
- 禁止空 `catch`
- 禁止“先吞异常再继续往下跑”

## 输入校验

默认要求：
- 在系统边界校验所有输入
- 能用 schema / 注解时优先使用
- 尽快失败，错误信息明确
- 默认不信任用户输入、外部接口返回、文件内容

## 代码质量自检

完成工作前，至少检查：

- [ ] 命名清晰，业务语义明确
- [ ] 函数足够短小，职责单一
- [ ] 文件足够聚焦
- [ ] 嵌套层级可控
- [ ] 错误处理完整
- [ ] 没有不必要的硬编码
- [ ] 没有明显共享可变状态问题

## Git 规范

推荐格式：

```text
<type>(<scope>): <description>

feat(auth): 添加登录功能
fix(task): 修复任务创建校验问题
refactor(business): 重构任务业务层
test(task): 添加任务服务单元测试
docs(readme): 更新说明
```

类型：

`feat` | `fix` | `refactor` | `perf` | `test` | `docs` | `style` | `build` | `chore`

分支命名：

`feature/{需求编号}-{功能描述}` | `fix/{问题描述}`

## 适用边界

这份规则适合作为所有项目的默认编码下限。

如果某个技术栈需要更细的约束，比如：
- Java DTO / Mapper 规范
- SQL / 索引规范
- 前端组件组织规范

应在对应技术栈规则中继续补充，而不是把所有细节都塞进本文件。
