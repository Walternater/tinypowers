# 代码审查清单

本文件提供一个通用的代码审查检查表。

它适合作为：
- 提交前自检清单
- 代码评审时的统一入口
- 子 Agent 输出审查报告时的补充参考

## 审查目标

代码审查至少要回答三件事：
- 这段实现有没有明显风险
- 这段实现能不能长期维护
- 这段实现有没有性能或一致性问题

## 第一优先级：阻塞问题

出现以下问题时，通常不应直接合并。

### 数据类型与契约一致性

- [ ] 请求类与实体类字段类型匹配
- [ ] 实体类与数据库字段类型兼容
- [ ] 跨层类型转换没有精度或语义丢失
- [ ] 时间类型使用一致

### 分页与查询正确性

- [ ] offset 计算正确
- [ ] `pageNum` 语义明确且一致
- [ ] 没有把 `pageNum` 直接当 offset 使用
- [ ] 分页参数有边界校验

### 枚举与状态完整性

- [ ] 枚举覆盖主要业务场景
- [ ] 枚举含义与需求一致
- [ ] 新增枚举不会破坏既有逻辑

### 安全问题

- [ ] SQL 使用参数化
- [ ] 日志已脱敏
- [ ] 关键接口具备权限校验

## 第二优先级：强烈建议修复

### 性能问题

- [ ] 没有明显全量加载风险
- [ ] 批量操作使用批量 SQL 或等价方案
- [ ] 统计逻辑优先交给数据库聚合
- [ ] 没有明显 N+1 查询

### 参数校验完整性

- [ ] 必填参数有 null 校验
- [ ] 空字符串和 null 被区分处理
- [ ] 长度与集合边界已校验
- [ ] 格式校验完整

### 防重复与幂等

- [ ] 关键唯一性有索引或等价保证
- [ ] 新增前有必要的重复校验
- [ ] 重复调用不会产生错误副作用

### 异常处理

- [ ] 异常信息明确
- [ ] 异常没有被静默吞掉
- [ ] 写操作失败时事务能正确回滚

## 第三优先级：可选优化

### 代码风格

- [ ] 注释风格统一
- [ ] 命名符合团队约定
- [ ] 类文件结构清晰

### 常量与魔法数字

- [ ] 反复出现的硬编码已抽常量
- [ ] 状态码和类型码使用枚举或常量
- [ ] 重复字符串已适当抽取

### 日志规范

- [ ] 日志级别合适
- [ ] 日志包含必要上下文
- [ ] 日志不暴露敏感信息

### 冗余代码

- [ ] 删除未使用的方法和变量
- [ ] 重复逻辑适当抽取
- [ ] 已有 Mapper / 公共方法被合理复用

## 提交前最小自检

| 检查项 | 状态 |
|--------|------|
| 代码能编译通过 | [ ] |
| 单元测试通过 | [ ] |
| 新增代码有基本测试覆盖 | [ ] |
| 无遗留 `TODO` / `FIXME` | [ ] |
| SQL 与枚举等关键实现已校对 | [ ] |

## 修复优先级建议

### 立即修

1. 数据类型或接口契约不一致
2. 分页实现错误
3. 枚举值缺失
4. SQL 注入或明显安全问题

### 尽快修

1. 性能问题
2. 参数校验缺陷
3. 缺少防重复或幂等设计
4. 异常处理不完整

### 视情况优化

1. 注释和命名风格
2. 魔法数字与硬编码
3. 日志表达
4. 代码冗余

## 和多 Agent 审查的关系

如果使用 tinypowers 的多阶段审查，建议把这份清单作为补充，而不是替代：

```text
Spec Compliance Review
  -> Security Review
  -> Code Review
```

也就是说：
- “实现是否符合方案”优先看方案符合性审查
- “有没有安全风险”优先看安全审查
- 这份清单更适合作为通用质量补充和提交前自查
