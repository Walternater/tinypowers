# 四.SQL类规范

#### 【建议】sql语句尽可能简单

> 1.mysql中一条SQL只能在一个CPU中执行，5000+ QPS的高并发中，运行1秒的大SQL意味着可能会堵死整个数据库
> 
> 2.拒绝大SQL，拆解成多条简单SQL
>    简单SQL缓存命中率更高
>    更适合多CPU环境执行

#### 【建议】保持事务（连接）短小

> 保持事务／DB连接短小精悍
> 
>     事务（连接）使用原则：即开即用，用完即关
> 
>     与事务无关操作放到事务外面，减少锁资源的占用时间
> 
>     不破坏一致性的前提下，使用多个短事务代替长事务
> 
> 反例：
>     上百个长时间sleep的连接

#### 【建议】尽可能的避免使用SP/TRIG/FUNCTION

> 1.线上OLTP系统（线下库另论）
>   尽可能少用存储过程
>   尽可能少用触发器
>   减少使用mysql函数对结果进行处理
> 
> 2.由客户端程序负责

#### 【建议】尽量不用select ＊

> 1.使用select ＊时
>    更多消耗CPU、内存、IO、网络带宽
>    会先像数据库请求所有列，然后去掉不需要列
> 2.尽量不用select ＊，只取需要的数据列
>    安全的设计：减少表变化带来的影响
>    为使用covering index提供可能性
>    select／join减少磁盘临时表生成。特别是有TEXT/BLOB时
> 举例：select \* from tag where id=99999; 修改为 select keyword1,keyword2 from tag where id=99999;

#### 【建议】改写or语句

> 1.同一字段，将or改写为in()
>   or效率：O(n)
>   in效率：O(Log n)
>   n很大时，or会慢很多
> 2.注意控制in的个数，建议n小于200
> 举例：
>   select \* from app where phone=’123456’ or phone=‘444444444’;
> 
>   改写为：
> 
>   select \* from app where phone in (’123456’,’654321’);
> 
> 3.不同字段，将or改为union
>   减少对不同字段进行or查询
> 举例：
>    select \* from app where phone=‘010-9999999’ or cellphone=‘1111111111’;
> 
>    改写为：
> 
>    select \* from app where phone=‘010-9999999’  union select \* from app where cellphone=‘1111111111’;

#### 【建议】避免负向查询和％前缀模糊查询

> 1.避免负向查询
>   not、！＝、<>、！<、!>、not exists、not in、not like等
> 2.避免％前缀模糊查询
>   B+ Tree使用不了索引，导致全表扫描
> 举例：
>    MySQL> select \* from post WHERE title like ‘北京%' ; 
>    298 rows in set (0.01 sec) 
>    MySQL> select \* from post WHERE title like '%北京%' ; 
>    572 rows in set (3.27 sec) 

#### 【建议】UPDATE、DELETE语句不使用LIMIT

> 在某些复制场景下，update、delete使用limit会导致主从数据不一致

  

#### 【建议】减少count(\*)

> innodb的count会扫描所有的结果集，资源开销大，尽量少用。

#### 【建议】limit高效分页

>             1.传统分页：
>                    select \* from table limit 1000,10;
>             2.limit原理：
>                    limit 10000,10
>                    偏移量越大越慢
>             3.推荐分页：
>                    select \* from table where id>=1234 limit 11;
>                    ＃10+1（每页10条）
>             4.分页方式二：
>                    select \* from table where id >=(select id from table limit 1000,1) limit 10;
>             5.分页方式三：
>                    select \* from table inner join (select id from table limit 1000,10) using (id);
>             6.分页方式四：
>                    程序取ID:select id from table limit 1000,10;
>                    select \* from table where id in (123456,……);
>             7.可能需要按场景分析并重组索引
>                    MySQL>select sql\_no\_cache \* from post limit 10,10’ > 	
> 
>                    10 row in set (0.01 sec) 
>                    MySQL> select sql\_no\_cache \* from post limit 20000,10; 
>                    10 row in set (0.13 sec) 
>                    MySQL> select sql\_no\_cache \* from post limit 80000,10; 
>                    10 rows in set (0.58 sec) 
>                    MySQL> select sql\_no\_cache id from post limit 80000,10; 
>                    10 rows in set (0.02 sec) 
>                    MySQL> select sql\_no\_cache \* from post WHERE id>=323423 limit 10; 	10 rows in set (0.01 sec) 
>                    MySQL> select \* from post WHERE id >= ( select sql\_no\_cache id from post limit 80000,1 ) limit 10 ; 
>                    10 rows in set (0.02 sec) 

#### 【建议】用union all而非union

> 若无需对结果进行去重，则用union all,因为union有去重开销

#### 【建议】分解连接保证高并发

> 1.高并发DB不建议进行两个表以上的join  
> 2.适当分解连接保证高并发
> 
> 举例：  
>       select name from tag join tag\_post on tag\_post.tag\_id=[tag.id](http://tag.id) join post on tag\_post.post\_id=post\_id where tag.tag=‘二手车’;  
>       改写为：    
>       select id from tag where ag=‘二手车’;  
>       select id from tag\_post where tag\_id=123;  
>       select name from post where post\_id in (123,456,789);

#### 【建议】group by优化

> 1.group by可以实现 分组，自动排序，没有特殊要求下，不用再进行排序操作
> 
> 2.尽量不要对大结果集进行group by操作，因为一旦数据量超过系统最大临时表大小时，mysql会将临时表里的数据copy到磁盘上然后再进行操作，性能会成数量级的下降

#### 【建议】同数据类型的列值比较

> 1.原则：数字对数字，字符串对字符串
> 
> 2.数值列与字符串列比较。需要同时转换为双精度
> 
> 3.字符串列与数值列比较，字符串列整列转为数值。并且不会使用索引查询
> 
> 4.举例：字符串列与数值列比较
>                    字段:\`remark\` varchar(50) NOT NULL COMMENT '备注, 默认为空', 
>                    MySQL>SELECT \`id\`, \`gift\_code\` FROM gift WHERE \`deal\_id\` = 640 AND remark=115127; 
>                    1 row in set (0.14 sec) > MySQL>SELECT \`id\`, \`gift\_code\` FROM pool\_gift WHERE 
>                    \`deal\_id\` = 640 AND remark='115127'; 1 row in set (0.005 sec) 

#### 【建议】尽量不用insert.......select语句

> 数据量大的情况下会造成主从延迟
> 在某些场景下会造成主从同步出错

#### 【建议】打散大批量更新

> 1.大批量更新凌晨操作，避开高峰
> 2.凌晨不限制
> 3.白天上限为100条／秒
> 举例：
>     update post set tag=1 where id in (1.2.3);
>     sleep 0.01;
>     update post set tag=1 where id in (4,5,6);
>     sleep 0.0.1;  
>
