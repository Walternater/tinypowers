# 一.核心规范

#### 【强制】 不要把聚合运算丢到数据库，让数据库多做它擅长的事情，MySQL不擅长做运算

> 1.复杂的运算移到程序端CPU，不要丢给数据库来做
> 
> 2.线上读写环境中使用order和group前要考虑是不是必须这么做，尽量减少没必要的排序和分组
> 
> 危害：MySQL是单进程操作，过多使用CPU来处理复杂的运算逻辑，会严重影响整个实例的性能
> 
> 反例：md5（）、order by rand（）都放到数据库中
> 
>      在MySQL中使用触发器来控制表之间数据的一致性
> 
>      使用存储过程来处理复杂的业务逻辑

#### 【强制】 拒绝3B（大SQL，大事物，大批量）

> 1.对于大批量的数据更新、修改、删除，严格控制每1000-2000条sleep1-2秒
> 
> 2.对于大事务，能拆分的情况下一定要拆分成多个小事物来处理，同时注意小事物的提交速度和间歇时间
> 
> 危害：大事务、大批量的数据变动，会造成主从同步延迟，在读写分离环境中非常影响业务读取的准确性
> 
> 反例：一条大sql 修改 5000行以上的数据。如：update t1 set name = 'bb' where is\_delete = 0; 
> 
>           无间歇的执行5000条以上变更没有任何sleep 。如：update t1 set name =  'aa' where id = xx; 连续循环5000次以上

#### 【强制】严格控制SQL语句效率

> 1.对于大表相关的SQL语句，上线前务必要确认是否能使用索引,在real环境执行计划，扫描行数在1W以上的必须谨慎，避免新上线业务爆发大批量的全表扫描SQL，最终影响整个实例的稳定性
> 
> 2.DBA分配的读域名都是有多个实例负载均衡的，所以查询类型的SQL尽量放在读域名去运行，避免所有的压力全部丢到主库单节点上
> 
> 3.对于一些业务上无法避免的统计类型大SQL，请联系DBA获取统计从库的连接信息，不要放在业务读写节点执行
> 
> 危害：大表的全表扫描会非常消耗磁盘IO，严重影响到同实例下所有的数据库性能以及相同文件系统下其他实例的性能
> 
> 反例：上线之前未审核大表相关SQL，出现一些全表扫描1000W行数据的高频SQL，导致数据库性能大降，连接数堆积，最终导致整个实例连接数爆满，无法连接

#### 【强制】严格控制连接数

> 1.对于MySQL5.7在我们的服务器上性能测试结果，整个实例的并发在120左右便达到最大的QPS
> 
> 2.并发在150之后随着连接数的增多，性能并不会得到提升。并且，在连接数超过500之后，出现性能严重下降
> 
> 3.请务必严格控制连接数，避免出现：连接数过高-->实例性能下降-->慢查询增多-->连接数持续堆积-->性能持续下降-->实例连接数爆满-->无法接受任何新的连接

#### 【强制】合理规划表大小，控制单表数据行数和列的个数

> 1.单表的行数尽量不要超过1000W行。没有大字段情况下，列数尽量不要超过50列，有大字段则需要控制列数更少
> 
> 2.对查询性能要求很高的表，尽量不要超过100W行
> 
> 3.大表建议进行分库分表，使用hash、取模等类似算法将数据散布在小表中
> 
> 4.日志类型的数据可以采用时间分区或者分表，表名后缀必须使用特定格式，比如按日散表user\_20110209、按月散表user\_201102
> 
> 5.能归档的数据，提前与DBA沟通定期归档策略（DBA进行自动归档）
> 
> 6.单库不超过200个表
