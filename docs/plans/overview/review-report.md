### 1.2 具体问题分析

#### REQ-001: 虚假依赖声明
**问题**: 1.1-1.5的多个任务声明"依赖: 无"
```markdown
### Task 1.1.1: 编译门禁脚本
- **依赖**: 无  ← 问题！实际依赖 Task 1.0.1.1 的 detect_stack 逻辑
```

**影响**: 若按此分配任务，独立开发的agent会重复实现或假设基础功能存在。

**修复建议**: 
- 明确标注依赖1.0的哪些基础组件
- 或定义"虚拟接口"，允许mock实现

## 5. 版本可并行性审查

### 5.1 当前依赖关系

```
当前（串行）:
1.0 (15天) → 1.1 (5天) → 1.2 (10天) → 1.3 (8天) → 1.4 (8天) → 1.5 (14天) → 2.0 (24天)

理论并行（若契约明确）:
                    ┌→ 1.1 (5天) ─┐
                    ├→ 1.2 (10天)─┤
1.0 (15天) ─────────┼→ 1.3 (8天) ─┼→ 1.5 (14天) → 2.0 (24天)
                    ├→ 1.4 (8天) ─┘
                    └→ (可并行)
```

### 5.2 并行开发 blocker

| Blocker | 影响版本 | 说明 |
|---------|----------|------|
| 缺少接口契约 | 1.1-1.4 | 不知道1.0会提供什么接口 |
| 缺少数据格式 | 1.2, 1.3 | 规则文件格式、覆盖率报告格式未约定 |
| 缺少扩展点 | 2.0 | 1.2的规则如何被2.0复用/重构未约定 |

### 5.3 启用并行开发的必要补充

需要创建以下**契约文档**：

1. **Interface Contract 1.0** (`contracts/v1.0-interface.md`)
   - 脚本输入/输出格式
   - 文件路径约定
   - 返回值规范

2. **Data Format Contract** (`contracts/data-formats.md`)
   - spec.md 格式规范（用于1.2审查）
   - coverage report 格式规范（用于1.3）
   - knowledge.md 格式规范（用于1.4）

3. **Extension Point Contract** (`contracts/extension-points.md`)
   - 规则如何被覆盖/扩展
   - 脚本钩子机制

## 7. 结论与建议

### 7.1 当前状态评估

- **1.0 MVP**: 文档较为完整，可独立开发，但需补充接口契约文档
- **1.1-1.4**: 可并行开发的前提是有明确的1.0接口契约
- **1.5**: 依赖前序版本的稳定性，无法并行
- **2.0**: 可部分并行（规则/模板定义），但需明确与1.x的兼容策略

### 7.2 推荐开发模式

**方案A：保守串行（当前）**
- 按 1.0 → 1.1 → ... → 2.0 顺序开发
- 风险低，但周期长（84天）

**方案B：契约先行并行（推荐）**
1. 先花费 **2-3天** 创建契约文档
2. 然后并行开发：
   - 团队A：1.0 (15天)
   - 团队B：1.1-1.4 准备（等1.0接口冻结后开始）
   - 团队C：2.0 架构设计（可提前）
3. 预计节省 **20-30%** 时间

### 7.3 下一步行动

1. **立即行动**：创建 `contracts/` 目录和基础契约文档
2. **本周内**：修复所有虚假依赖声明
3. **下周前**：补充量化验收标准
4. **1.0启动前**：完成接口契约冻结
