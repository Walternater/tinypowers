# 2.0 执行层生态与智能化详细设计

## 执行层生态

### 支持的执行引擎

| 引擎 | 类型 | 核心能力 | 适用场景 |
|------|------|----------|----------|
| superpowers | 默认 | brainstorming, writing-plans, subagent-driven-development | 全流程 |
| get-shit-done | 执行 | wave-execution, deviation-control | 大feature分波执行 |
| gstack | 审查 | ceo-review, eng-review, design-review, qa, cso | 多维度专业审查 |
| openspec | 规划 | propose, apply, archive | 快速路径 |
| cc-sdd | 全栈 | spec-init, spec-requirements, spec-design, spec-tasks, spec-impl | EARS格式需求 |

### 引擎适配器接口

```typescript
interface EngineAdapter {
  name: string;
  type: 'default' | 'execution' | 'review' | 'planning' | 'full-stack';
  
  // 能力声明
  capabilities(): Skill[];
  
  // 执行任务
  execute(skill: string, input: any): Promise<Output>;
  
  // 输出格式转换
  normalize(output: any): StandardOutput;
}
```

### 调度策略

```yaml
# 默认调度
/tech:feature:
  planning_engine: superpowers  # 默认
  
# 快速路径（小需求）
/tech:feature:
  planning_engine: openspec
  
# 复杂需求（多角色审查）
/tech:feature:
  planning_engine: gstack
  
# 大feature分波执行
/tech:code:
  execution_engine: get-shit-done
  config:
    waves: 3
    deviation_threshold: 0.8
```

## 智能化增强

### AI自动修复

#### 修复场景矩阵

| 问题类型 | 检测来源 | 自动修复 | 需确认 |
|----------|----------|----------|--------|
| 命名规范 | compliance-reviewer | ✅ 重命名 | |
| 格式化 | check-style | ✅ spotless:apply | |
| 缺失import | compile error | ✅ 自动添加 | |
| 简单安全漏洞 | security-check | | ✅ 添加校验注解 |
| 测试骨架 | coverage check | ✅ 生成 | |
| 复杂逻辑 | code-review | | ✅ 仅建议 |

#### 修复流程

```
检测到问题
    ↓
查询修复模板
    ↓
生成修复方案
    ↓
应用修复到临时文件
    ↓
编译验证
    ↓
测试验证
    ↓
通过 → 提交修复
失败 → 回滚，转人工
```

### 智能建议

#### 建议分类

**架构建议**
- 重复代码检测 → 提取公共方法
- 循环依赖检测 → 重构包结构
- 上帝类检测 → 职责拆分

**性能建议**
- N+1查询检测 → 建议使用JOIN
- 同步阻塞检测 → 建议使用异步
- 大数据量检测 → 建议添加分页

**安全建议**
- 敏感信息日志 → 建议脱敏
- 弱加密算法 → 建议升级
- 缺失鉴权 → 建议添加检查

**最佳实践**
- 旧API使用 → 建议迁移
- 缺失文档 → 建议添加
- 低覆盖率 → 建议补充测试
