# tinypowers 1.0 设计文档

## 设计决策汇总

| 决策项 | 选择 | 理由 |
|--------|------|------|
| 核心价值 | 强制先规划后编码 + 标准化交付物 | 解决AI跳步问题，统一协作语言 |
| 门禁强度 | 硬门禁 | 工具必须有约束力 |
| 交付物数量 | 3个（PRD/spec/tasks） | 覆盖规划必需，不冗余 |
| 技术栈 | Java only | 先深度后广度 |
| AI介入 | 引导式问答 | 平衡效率与质量 |
| 代码审查 | 专用agent | 保证审查深度 |
| 状态管理 | 简单DONE标记 | 不增加管理负担 |

## 三阶段流程

### 阶段1：/tech:feature（规划）

**目标**：产出可直接指导编码的规划包

**流程**：
```
1. 用户输入 /tech:feature {需求编号} {需求简述}
2. AI 检查是否存在 features/{id}/ 目录
   - 存在 → 读取已有文档继续编辑
   - 不存在 → 创建目录，初始化模板
3. AI 通过问答引导用户明确需求（5-8个问题）
4. AI 生成 PRD.md / spec.md / tasks.md
5. 用户审核确认或要求修改
6. 确认后标记为"PLAN DONE"
```

**AI提问示例**：
- 这个需求的背景是什么？解决什么问题？
- 目标用户是谁？在什么场景下使用？
- 范围边界：包含什么？明确排除什么？
- 验收标准：怎样算做完了？
- 技术约束：必须用什么技术？不能用什么？

**交付物标准**：

| 文件 | 必须包含 | 门禁检查 |
|------|---------|---------|
| PRD.md | 背景、范围边界、≥3条验收标准 | 非空 |
| spec.md | 目标、核心设计、≥1条锁定决策 | 非空且有决策 |
| tasks.md | 任务列表（≤8个）、验收标准 | 非空且可执行 |

---

### 阶段2：/tech:code（编码）

**目标**：按规划完成实现，通过审查

**硬门禁（进入条件）**：
```markdown
□ PRD.md 存在且非空
□ spec.md 存在且有≥1条锁定决策  
□ tasks.md 存在且任务≤8个
□ 以上不满足 → AI必须拒绝编码请求
```

**流程**：
```
1. 用户输入 /tech:code
2. AI 执行门禁检查
   - 失败 → 提示缺少什么，要求先完成 /tech:feature
   - 通过 → 继续
3. AI 读取当前任务，询问用户从哪个任务开始
4. 按任务执行编码
5. 全部任务完成后，执行决策自查：
   - 逐条核对 spec.md 中的锁定决策
   - 确认每条决策都有对应代码实现
6. 调用 reviewer agent 执行代码审查
7. 输出审查结果，如有问题要求修复
8. 通过后标记为"CODE DONE"
```

**审查输出格式**：
```markdown
## 代码审查结果

### BLOCK（必须修复）
- [ ] 问题描述 → 修复建议

### WARN（建议修复）  
- [ ] 问题描述 → 修复建议

### PASS（通过）
- [x] 检查项说明

**结论**: BLOCK 0 / WARN 2 / PASS 15 → 通过/不通过
```

---

### 阶段3：/tech:commit（完成）

**目标**：提交代码，完成交付

**硬门禁（进入条件）**：
```markdown
□ 代码编译通过
□ 决策自查完成（所有锁定决策已落地）
□ 代码审查无BLOCK级问题
□ 以上不满足 → AI必须拒绝提交
```

**流程**：
```
1. 用户输入 /tech:commit
2. AI 执行门禁检查
   - 失败 → 提示缺少什么
   - 通过 → 继续
3. 执行最终检查：
   - 代码与文档一致
   - 无debug代码残留
   - 无敏感信息泄露
4. 生成 commit message
5. 创建 DONE 标记文件
6. 提示用户手动执行 git commit/push
```

**DONE标记**：
```markdown
# DONE

- 需求编号: {id}
- 完成时间: {date}
- 交付物:
  - [x] PRD.md
  - [x] spec.md  
  - [x] tasks.md
  - [x] 代码实现
  - [x] 审查通过
```

## 核心设计原则

### 1. 硬门禁设计

门禁必须在AI层面强制执行，而不是仅靠用户自觉。

```javascript
// 伪代码示例
defineCommand('/tech:code', () => {
  const planFiles = checkPlanFiles();
  if (!planFiles.valid) {
    return `❌ 门禁未通过：缺少 ${planFiles.missing}
            请先完成 /tech:feature 阶段`;
  }
  // 继续编码...
});
```

### 2. 引导式问答

AI不直接生成完整文档，而是通过提问帮助用户理清思路。

原则：
- 一次问一个问题
- 优先选择题降低思考负担
- 基于回答追问，深挖关键细节

### 3. YAGNI（你不会需要它）

1.0 明确不包含：
- [ ] MySQL规则（非核心）
- [ ] 多技术栈支持（后续扩展）
- [ ] SPEC-STATE状态机（简化标记足够）
- [ ] hooks运行时守护（AI层门禁足够）
- [ ] 自动化脚本（手动执行git）
- [ ] 安装工具（手动复制文件）
