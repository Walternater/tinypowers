### 1.0.1 /tech:init 实现

**功能**：技术栈检测 + 骨架初始化

**具体实现**：
```markdown
## /tech:init SKILL.md

### 触发条件
- 新项目首次使用
- 用户主动调用 /tech:init

### 执行流程
1. 检测技术栈
   - 检查 pom.xml → Maven
   - 检查 build.gradle → Gradle
   - 检查 src/main/java → Java 辅助

2. 确认信息（询问用户）
   - 项目名称
   - 技术栈确认
   - 构建命令确认

3. 生成骨架
   - CLAUDE.md
   - docs/knowledge.md（骨架）
   - features/ 目录

4. 输出完成确认
```

**技术栈检测伪代码**：
```bash
detect_stack() {
  if [ -f "pom.xml" ]; then
    echo "java-maven"
    BUILD_COMMAND="mvn test"
  elif [ -f "build.gradle" ]; then
    echo "java-gradle"  
    BUILD_COMMAND="./gradlew check"
  else
    echo "ERROR: 未检测到 Java 项目"
    exit 1
  fi
}
```

**验收标准**：
- [ ] 检测到 pom.xml 生成 Maven 配置
- [ ] 检测到 build.gradle 生成 Gradle 配置
- [ ] 生成 CLAUDE.md 包含正确构建命令
- [ ] 非 Java 项目给出明确错误提示

**时间**：3天

### 1.0.2 /tech:feature 实现

**功能**：需求规划 + CHECK-1 门禁

**具体实现**：
```markdown
## /tech:feature SKILL.md

### 触发条件
- 开始新需求
- 规划阶段未完成

### 执行流程
1. 输入处理
   - 读取用户输入：/tech:feature {id} {描述}
   - 检查 features/{id}/ 是否存在
   - 存在 → 读取继续编辑
   - 不存在 → 创建目录，初始化模板

2. 引导式问答（5-8个问题）
   Q1: 这个需求的背景是什么？
   Q2: 目标用户是谁？在什么场景使用？
   Q3: 范围边界：包含什么？明确排除什么？
   Q4: 验收标准：怎样算做完了？
   Q5: 技术约束：必须用什么？不能用什么？

3. 委托 superpowers brainstorming
   - 输入：问答结果
   - 输出：方案探索结果

4. 委托 superpowers writing-plans
   - 输入：方案探索结果
   - 输出：PRD.md / spec.md / tasks.md

5. 用户确认
   - 展示生成文档
   - 询问是否修改
   - 确认后标记 PLAN DONE

6. CHECK-1 门禁检查
   - 自动检查交付物完整性
   - 输出 CHECK-1 报告
```

**CHECK-1 门禁定义**：
```markdown
## CHECK-1 (feature → code)

□ PRD.md 存在且非空
  - 背景说明（≥1段）
  - 范围边界（包含/排除）
  - 验收标准（≥3条，EARS格式）

□ spec.md 存在且非空
  - 目标与范围
  - 核心设计
  - 锁定决策（≥1条，D-XXX格式）

□ tasks.md 存在且非空
  - 任务列表（≤8个）
  - 每个任务有验收标准
  - 依赖关系清晰

结论：PASS / FAIL
- PASS：可进入 /tech:code
- FAIL：列出缺失项，要求补充
```

**交付物模板**：

PRD.md：
```markdown
# PRD: {需求名称}

## 背景
{为什么做这个需求}

## 范围
### 包含
- {功能点1}
- {功能点2}

### 排除
- {明确不做的事}

## 验收标准
- [ ] AC-001: Given {上下文} When {动作} Then {结果}
- [ ] AC-002: ...
- [ ] AC-003: ...
```

spec.md：
```markdown
# 技术方案: {需求名称}

## 目标
{要达成什么}

## 核心设计
{架构描述}

## 锁定决策
| ID | 决策 | 理由 |
|----|------|------|
| D-001 | {关键决策} | {为什么} |
```

tasks.md：
```markdown
# 任务拆解: {需求名称}

| ID | 任务 | 验收标准 | 依赖 |
|----|------|---------|------|
| T-001 | {任务1} | {标准} | - |
| T-002 | {任务2} | {标准} | T-001 |
```

**验收标准**：
- [ ] 能引导用户回答5-8个问题
- [ ] 正确委托 brainstorming 和 writing-plans
- [ ] CHECK-1 不满足时拒绝进入 /tech:code
- [ ] 输出格式符合模板规范

**时间**：5天

### 1.0.3 /tech:code 实现

**功能**：编码执行 + CHECK-2 门禁 + compliance-reviewer

**具体实现**：
```markdown
## /tech:code SKILL.md

### 触发条件
- CHECK-1 已通过
- SPEC-STATE 为 PLAN
- 用户调用 /tech:code

### 执行流程

#### Phase 1: CHECK-2 进入门禁
自动检查：
□ CHECK-1 已通过
□ PRD/spec/tasks 存在且有效
□ 无进行中的其他 feature（可选）

不通过 → 提示缺少什么，要求先完成 feature 阶段
通过 → 继续

#### Phase 2: Pattern Scan（缝合扫描）
扫描项目既有模式：
- Controller 命名/注解风格
- Service 事务使用方式
- Repository 继承关系
- 异常处理模式
- 日志使用方式

输出：patterns.md（供 superpowers 参考）

#### Phase 3: 委托 superpowers 执行

1. using-git-worktrees
   - 创建隔离 worktree
   - 返回 worktree 引用

2. subagent-driven-development
   - 输入：tasks.md + patterns.md
   - 执行：按任务编码
   - 输出：代码实现

3. 决策自查（开发者确认）
   逐条确认 D-XXX 决策已落地：
   □ D-001: {决策内容} → {代码位置}
   □ D-002: ...

4. compliance-reviewer（tinypowers 独有）
   审查维度：
   - 决策落地：每条决策有代码体现
   - 接口符合：API 与方案一致
   - 数据符合：DB 变更与方案一致
   - 范围符合：无方案外变更
   - 安全符合：无新增风险

   输出：BLOCK / WARN / PASS

5. requesting-code-review（superpowers）
   审查维度：
   - 代码质量
   - 设计模式
   - 性能隐患

6. verification-before-completion（superpowers）
   验证维度：
   - 功能符合验收标准
   - 测试通过

#### Phase 4: CHECK-2 离开门禁

□ 代码编译通过（人工确认）
□ compliance-reviewer 通过（无 BLOCK）
□ requesting-code-review 通过（无 BLOCK）
□ verification-before-completion 通过
□ 决策自查完成

不通过 → 要求修复，重新审查
通过 → 生成 VERIFICATION.md，标记 CODE DONE

#### Phase 5: 输出
- 代码实现（在 worktree）
- VERIFICATION.md
- 审查报告汇总
```

**compliance-reviewer.md**（agents/）：
```markdown
# compliance-reviewer

## 职责
审查代码实现是否符合技术方案

## 输入
- spec.md（技术方案）
- 代码变更（diff）

## 审查清单

### 决策落地（逐条核对）
对于每个 D-XXX 决策：
□ 代码中有对应实现
□ 实现方式与决策一致
□ 有注释或命名体现

### 接口符合
□ API 路径与方案一致
□ 请求/响应参数与方案一致
□ 状态码使用正确

### 数据符合
□ DB 变更与方案一致
□ 字段类型/约束正确
□ 索引按方案创建

### 范围符合
□ 无方案外代码变更
□ 无未记录的附加功能
□ 无调试代码残留

### 安全符合
□ SQL 使用参数化查询
□ 用户输入有校验
□ 无敏感信息硬编码
□ 无越权访问风险

## 输出格式
```
## Compliance Review 结果

### 决策落地
- [x] D-001: 已落地 → OrderController.java:45
- [ ] D-002: 未找到对应实现 ⚠️

### 接口符合
- [x] API 路径符合
- [x] 参数定义符合

### 数据符合
- [x] DB 变更符合

### 范围符合
- [x] 无范围外变更

### 安全符合
- [x] 无安全风险

## 结论
**BLOCK**: 1 / **WARN**: 0 / **PASS**: 4

必须修复后才能提交。
```

**验收标准**：
- [ ] CHECK-2 进入门禁强制执行
- [ ] Pattern Scan 输出有效模式
- [ ] compliance-reviewer 能发现方案偏离
- [ ] BLOCK 级问题阻断流程
- [ ] CHECK-2 离开门禁强制执行

**时间**：5天

### 1.0.4 /tech:commit 实现

**功能**：文档同步 + Knowledge Capture + Git 提交

**具体实现**：
```markdown
## /tech:commit SKILL.md

### 触发条件
- CHECK-2 已通过
- SPEC-STATE 为 CODE DONE
- 用户调用 /tech:commit

### 执行流程

#### Phase 1: 前置检查
□ SPEC-STATE 为 CODE DONE
□ VERIFICATION.md 存在且结论为 PASS
□ 工作区无未提交的无关变更

不通过 → 提示问题
通过 → 继续

#### Phase 2: 文档同步检查
逐项确认：
□ 技术方案.md 中的接口与实际一致
□ VERIFICATION.md 结论正确
□ docs/knowledge.md 已更新（如有新知识）
□ README.md 已同步（如接口有变化）

不一致 → 提示更新
一致 → 继续

#### Phase 3: Knowledge Capture（独有）

判断是否有价值：
□ 发现了项目特有约定
□ 确认了框架用法约束
□ 找到了可复用模式
□ 踩到了隐蔽的坑

有价值 → 追加到 docs/knowledge.md
无价值 → 跳过

#### Phase 4: 生成提交信息

格式：
```
[AI-Gen] feat(scope): {description}

- {变更点1}
- {变更点2}

Verification: {验证结果}
```

#### Phase 5: Git 提交
1. 创建最终 commit（包含代码、文档、SPEC-STATE）
2. 提示用户执行：
   ```bash
   git push origin {branch}
   # 然后创建 PR
   ```

#### Phase 6: 委托 finishing-branch（可选）
用户确认后，委托 superpowers 清理分支

#### Phase 7: 标记 DONE
更新 SPEC-STATE → DONE
```

**Knowledge Capture 逻辑**：
```markdown
## 知识提取规则

触发时机：/tech:commit 阶段

提取来源：
1. spec.md 中的锁定决策
2. code review 中的关键讨论
3. 本次 feature 的特殊处理

写入格式（docs/knowledge.md）：
```markdown
## {日期} - {需求编号}

### 约定
**主题**: {主题}
**约束**: {具体约束}
**场景**: {什么时候用}
**来源**: {需求编号}

### 踩坑
**问题**: {问题描述}
**原因**: {为什么}
**解决**: {怎么办}
**来源**: {需求编号}
```
```

**验收标准**：
- [ ] 文档不一致时提示更新
- [ ] 有价值知识自动沉淀
- [ ] 生成符合规范的 commit message
- [ ] 成功标记 DONE

**时间**：2天
