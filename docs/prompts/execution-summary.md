# 执行摘要提示词

> 在完成一次完整的 init → feature → code → commit 演练后，用以下提示词请 AI 输出标准化执行摘要。

---

## 提示词

```
请按以下结构，以 Markdown 格式输出本次执行摘要：

## 完成度

用表格列出每个执行阶段的状态和产物：

| 阶段 | 状态 | 产物 |
|------|------|------|
| Worktree 创建 | ✅/⚠️/❌ | ... |
| tech:init | ✅/⚠️/❌ | ... |
| tech:feature | ✅/⚠️/❌ | ... |
| tech:code | ✅/⚠️/❌ | ... |
| tech:commit | ✅/⚠️/❌ | ... |
| 其他自定义阶段（如有） | ✅/⚠️/❌ | ... |

末尾注明：**完成度：X/X 项，XX%**

---

## 执行时长估算

按阶段拆分工具调用轮次和时间估算：

| 阶段 | 工具调用轮次 | 估计耗时 |
|------|------------|---------|
| ... | ~N 轮 | ~N min |
| **合计** | **~N 轮** | **~N min** |

---

## 关键数字

用简洁的 bullet 列出本次执行的关键指标，例如：
- 代码文件数（新增/修改）
- 文档文件数
- 测试用例数及通过率
- Git commit 数
- 发现问题数（如有审查环节）

> 输出格式要求：整份摘要使用标准 Markdown，表格、标题、bullet 严格按上述结构，不输出额外解释文字。
```

---

## 字段说明

| 字段 | 说明 |
|------|------|
| 阶段状态 | ✅ 完成 / ⚠️ 部分完成或有偏差 / ❌ 未完成或失败 |
| 产物 | 该阶段实际输出的文件、提交、结论等，用简短文本描述 |
| 工具调用轮次 | 大致估算，用于衡量执行代价；无精确时间戳时以轮次推算时长 |
| 估计耗时 | 按每轮约 15-30s 估算，精度为分钟级 |
| 关键数字 | 仅列出有意义的指标，不堆砌数字 |

---

## 使用场景

- 完成一次需求演练后，作为 session 收口的标准输出
- 多次演练对比时，作为横向比较基准
- 提交 PR 前，作为 reviewer 快速了解本次执行情况的入口

## 配套说明

- 本提示词不替代 `VERIFICATION.md`，后者关注"功能是否正确"，本摘要关注"执行过程的效率和完整性"
- 若有 workflow 审查需求，摘要之后可追加"步骤审查"部分（参考 `docs/workflow-review/` 目录的格式）
