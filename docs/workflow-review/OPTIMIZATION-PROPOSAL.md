# Tech Workflow 优化方案

> 基于完整执行 tech:init → tech:feature → tech:code → tech:commit 流程的实际体验

## 执行摘要

| 阶段 | 耗时 | 主要痛点 |
|------|------|---------|
| tech:init | 5 min | 输出文件较多，部分规则可能用不上 |
| tech:feature | 20 min | 文档格式要求严格，内容有重复 |
| tech:code | 30 min | SPEC-STATE 推进繁琐，格式校验严格 |
| tech:commit | 10 min | VERIFICATION 格式敏感，两阶段提交 |
| **总计** | **~65 min** | **流程复杂度高，知识沉淀不足** |

---

## 1. 核心问题诊断

### 1.1 文档爆炸
- **现象**: feature 阶段产出 6 个文档文件（PRD、技术方案、任务拆解、SPEC-STATE、测试计划、测试报告、VERIFICATION）
- **影响**: 信息分散，维护成本高，容易流于形式
- **根因**: 文档职责划分过细，存在内容重叠

### 1.2 格式束缚
- **现象**: AC 必须用 SHALL/EARS 格式，决策表格必须4列，VERIFICATION 不能用 Markdown 强调
- **影响**: 写文档时频繁被脚本打断，需要反复调整格式
- **根因**: 脚本校验过于严格，不支持自然语言变体

### 1.3 状态管理割裂
- **现象**: 每次状态推进需要运行 update-spec-state.js 脚本
- **影响**: 打断开发流，手工操作容易出错
- **根因**: 状态机与文档分离，没有自动化触发

### 1.4 知识沉淀失效
- **现象**: knowledge.md 初始为空模板，learnings.md 很少被填充
- **影响**: 项目知识无法有效积累
- **根因**: 缺乏知识提取的触发机制和示例引导

## 2. 优化方案

### 2.1 文档合并策略

#### 当前 vs 优化

| 当前文档 | 优化后 | 说明 |
|---------|--------|------|
| PRD.md | **需求设计.md** | 合并 PRD + 技术方案，业务+技术一体 |
| 技术方案.md | （合并） | 不再独立 |
| 任务拆解表.md | **任务拆解表.md** | 保留，但简化格式 |
| 测试计划.md + 测试报告.md + VERIFICATION.md | **验证报告.md** | 三合一，轻量级 |
| SPEC-STATE.md | **状态标记** | 内嵌到需求设计.md 头部 |
| learnings.md | **动态创建** | 有内容时才创建 |

#### 新的 Feature 目录结构
```
features/F001-order-management/
├── 需求设计.md          # 合并 PRD + 技术方案 + SPEC-STATE
├── 任务拆解表.md         # 简化版任务列表
├── 验证报告.md           # 合并测试计划/报告/VERIFICATION
└── .assets/              # 可选：截图、流程图等
```

### 2.2 简化格式校验

#### AC（验收标准）格式放宽
```markdown
; 当前（严格）
**AC-1**: WHEN 用户提交订单 THEN 系统 SHALL 在 500ms 内返回结果。

; 优化后（支持自然语言）
AC1: 用户提交订单时，系统应在 500ms 内返回结果。
AC2: [性能] 订单查询接口 P99 < 100ms。
```

#### 决策记录简化
```markdown
; 当前（4列表格）
| 编号 | 决策项 | 理由 | 状态 |
| D-1 | 使用 Seata AT | 低侵入 | 已确认 |

; 优化后（清单式）
## 关键决策
- [x] 分布式事务：采用 Seata AT 模式（低侵入、成熟）
- [x] 状态机：使用 Spring Statemachine（可控、可扩展）
```

#### 验证结论简化
```markdown
; 当前（纯文本）
- Result: PASS

; 优化后（支持 Markdown）
**验证结论**: ✅ PASS
```

### 2.3 自动化状态流转

#### 当前流程
```
开发完成 → 运行脚本 → EXEC→REVIEW → 运行脚本 → REVIEW→DONE
```

#### 优化后流程
```
开发完成 → 自动检测 → 状态更新
```

**实现方式**:
- 在需求设计.md 头部用简单标记表示状态：`Status: EXEC`
- 编码完成后自动读取 git 状态更新标记
- 不再依赖外部脚本推进状态

### 2.4 知识沉淀改进

#### 自动知识提取
```javascript
// 开发完成后，自动从以下位置提取知识：
1. 代码中的注释（@pattern, @pitfall 等标签）
2. 需求设计.md 中的"踩坑记录"区块
3. 验证报告.md 中的"遗留事项"
```

#### knowledge.md 结构优化
```markdown
# 项目知识库

## 设计模式
<!-- 从代码注释自动提取 -->

## 已知问题
<!-- 从验证报告自动汇总 -->

## 最佳实践
<!-- 人工维护 + 代码审查沉淀 -->
```

### 2.5 Fast/Medium/Standard 路径优化

#### 当前三档差异不清晰
| 路径 | 差异点 |
|------|--------|
| Fast | 不明确 |
| Medium | 不明确 |
| Standard | 不明确 |

#### 优化后清晰定义

**Fast 路径**（< 2小时，单文件变更）:
- 只需：需求设计.md（1页以内）
- 跳过：详细任务拆解、正式测试报告
- 验证：代码审查 + 单元测试通过即可

**Medium 路径**（1-3天，模块级变更）:
- 需要：需求设计.md + 任务拆解表.md
- 简化：验证报告.md（轻量级）
- 验证：集成测试 + 代码审查

**Standard 路径**（> 3天，跨系统/架构级）:
- 完整：所有文档
- 可选：使用 worktree、subagent
- 验证：完整测试计划 + 多轮审查
