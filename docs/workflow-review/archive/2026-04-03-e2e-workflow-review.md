## 逐步审查

### 1. /tech:init

**执行过程**：运行 `init-project.js` → 验证通过 → 产出 CLAUDE.md + rules + guides + hooks + knowledge.md 模板

**复杂度评价**：低 ✅

**做得好的地方**：
- 初始化完全脚本化，无需 AI 手工操作
- 技术栈检测准确（Java Maven + MySQL）
- 内置验证通过，无需额外校验
- Lazy mode 知识扫描合理（空项目不强做）

**发现的问题**：

| # | 问题 | 严重度 | 说明 |
|---|------|--------|------|
| I-01 | 框架副本被完整复制到目标项目 | 高 | `install.sh` 把整个 `.claude/skills/tinypowers/` 复制进目标项目，包含 145+ 个文件（agents、docs/archive、所有 templates），导致后续 commit 全部包含这些文件 |
| I-02 | `target/` 目录无 `.gitignore` | 中 | 构建产物被提交到 git 历史 |
| I-03 | init 后无 `.gitignore` 创建 | 中 | 目标项目缺少基础的 `.gitignore` |

**合理性判断**：init 本身合理且高效，但**安装机制导致的目标项目膨胀**是最大问题。

---

### 2. /tech:feature

**执行过程**：创建 PRD → 需求理解 → 技术方案（4 条锁定决策） → 任务拆解表（10 任务） → SPEC-STATE

**复杂度评价**：中偏高 ⚠️

**做得好的地方**：
- 需求路由选择准确（Medium：10 任务，有 DB 变更，无跨系统依赖）
- 技术方案包含 4 条明确的锁定决策（D-001 到 D-004）
- 任务拆解有依赖关系和执行批次
- 验收标准使用 EARS 格式（When/Then）

**发现的问题**：

| # | 问题 | 严重度 | 说明 |
|---|------|--------|------|
| F-01 | SPEC-STATE 模板格式与脚本解析不匹配 | 高 | 脚本要求 `phase: PLAN` 在 YAML 代码块中 + `track` 字段，但默认创建时缺少这些，导致 `update-spec-state.js` 报错 "缺少合法的 phase" |
| F-02 | PRD 验收标准格式门禁过严 | 中 | 脚本要求 AC-N: 或 EARS 格式正则匹配，普通中文描述被拒绝。用户需要知道这个隐含格式要求 |
| F-03 | 技术方案"锁定决策"关键词硬编码 | 中 | 脚本检查 `/(已锁定决策\|决策记录\|锁定决策)/` 和 `\| 已确认 \|` 表格格式，文档作者需要知道这个精确格式 |
| F-04 | 需求理解.md 非标准产物 | 低 | SKILL.md 默认骨架不包含此文件，但实际分析时创建了，增加了文档数量 |
| F-05 | 无用户确认环节 | 中 | 流程设计有"方案与任务确认"步骤，但实际执行中 AI 自行完成，没有真正的暂停确认点 |

**合理性判断**：规划质量高，但**门禁脚本的格式要求与文档模板之间存在 gap**，导致需要反复修正格式才能通过。

### 3. /tech:code

**执行过程**：Gate Check → 推进 SPEC-STATE 到 EXEC → 委托 deep agent 实现 → 产出 23 个源码文件 → 编译通过 → 测试因环境失败 → 手动创建测试计划/报告/VERIFICATION → 推进到 REVIEW

**复杂度评价**：高 ⚠️

**做得好的地方**：
- Gate Check 门禁有效（验证了 PRD、技术方案、任务拆解的存在性和内容质量）
- 生成的代码质量不错：构造函数注入、@Transactional、BigDecimal 金额、遵循 4 条锁定决策
- 包结构符合 Java 编码规范
- SPEC-STATE 阶段推进正确

**发现的问题**：

| # | 问题 | 严重度 | 说明 |
|---|------|--------|------|
| C-01 | Deep agent 超时（30 min / 1800s） | 高 | 10 个任务的实现委托给 deep agent，超过最大等待时间。虽然最终产出了代码，但超时说明任务粒度或 prompt 有问题 |
| C-02 | 无审查环节 | 高 | SKILL.md 要求 compliance-reviewer → code-review 顺序，但实际未执行任何审查。直接跳到了测试验证 |
| C-03 | 测试计划/报告/VERIFICATION 手动创建 | 中 | 这些应该是执行过程的产物，但实际是编码完成后手动补写的"事后文档" |
| C-04 | 测试因 Java 版本不匹配失败 | 中 | pom.xml 未指定 Java 版本，测试类被编译为 Java 17 (class 61.0)，但运行环境是 Java 8。这是 init 时应该检测的 |
| C-05 | 无 compliance-reviewer 调用 | 高 | 技术方案中的 4 条决策是否在代码中被正确遵守，没有经过显式验证 |
| C-06 | 决策合规性在 VERIFICATION.md 中手动填写 | 低 | 应该是自动化检查，而非人工填写表格 |

**合理性判断**：代码实现质量可接受，但**审查环节完全缺失**，测试文档是"补作业"而非执行过程的自然产物。

## 优化方案

### 阶段一：立即可做（零代码改动）

#### 1.1 修复 .gitignore 缺失
```
init-project.js 在初始化时自动创建 .gitignore，至少包含：
- target/
- *.class
- *.jar
- .DS_Store
```

#### 1.2 SPEC-STATE 模板修复
```
确保 update-spec-state.js 能正确处理模板格式：
- 方案 A：让脚本同时支持 "当前阶段: PLAN" 和 "phase: PLAN" 两种格式
- 方案 B：更新 spec-state.md 模板确保包含 YAML 代码块
```

#### 1.3 PRD 验收标准格式提示
```
在 PRD 模板中增加注释说明：
"验收标准请使用 AC-N: 描述 或 EARS 格式（When...Then...）"
```

### 阶段二：Skill 层优化

#### 2.1 /tech:code 审查环节自动化

**当前问题**：compliance-reviewer 和 code-review 完全缺失。

**优化方案**：
```
在 /tech:code 的"审查修复"步骤中：
1. 编码完成后自动触发 compliance-reviewer（检查方案符合性 + 决策合规）
2. 自动检查锁定决策（D-001 到 D-004）是否在代码中被遵守
3. 审查结果自动写入 VERIFICATION.md 的"决策合规性"部分
4. 仅在审查发现严重问题时要求人工介入
```

#### 2.2 /tech:code 测试文档前置

**当前问题**：测试计划/报告是事后补写。

**优化方案**：
```
将测试计划创建移到 Gate Check 之后、开发执行之前：
1. Gate Check 通过后，基于任务拆解表自动生成测试计划初稿
2. 编码完成后执行测试（或跳过并记录原因）
3. 测试报告在执行时自动填写，而非事后补写
```

#### 2.3 /tech:feature 确认环节实质化

**当前问题**：确认步骤是 AI 自行完成，没有真正的暂停。

**优化方案**：
```
在输出技术方案和任务拆解表后，向用户输出确认摘要：
"方案已就绪，包含 N 条决策和 M 个任务。确认要点：
1. 技术方案是否覆盖所有需求？
2. 任务拆解是否有遗漏？
确认后输入 '继续' 进入 /tech:code"
```

### 阶段三：架构层优化

#### 3.1 框架安装机制重构（最关键）

**当前问题**：install.sh 把整个框架仓库复制到目标项目。

**优化方案**：
```
方案 A（推荐）：全局安装
- install.sh --global 安装到 ~/.claude/skills/tinypowers/
- 目标项目只安装 hooks + rules + CLAUDE.md 模板
- skills 通过全局路径引用，不复制到项目内

方案 B：精简安装
- install.sh 只复制运行时需要的文件（skills/*.md, hooks/*.js, rules/**）
- 不复制 docs/archive、manifests、scripts（除运行时需要的）
- 不复制 .claude/skills/tinypowers/ 下的框架自身文档
```

**影响**：
- 提交文件数：178 → ~35（减少 80%）
- 提交大小：~500KB → ~100KB（减少 80%）
- git 历史清洁度：显著提升

#### 3.2 知识沉淀链路激活

**优化方案**：
```
在 /tech:commit 的"交付后可选动作"中：
1. 检查 notepads/learnings.md 是否有内容
2. 如果有，自动提取到 docs/knowledge.md
3. 如果没有，检查本次 feature 是否有新的踩坑经验
4. 如果有，建议创建 learnings.md 条目

触发条件（至少满足一条）：
- 遇到了非显而易见的 bug 并解决
- 发现了框架自身的限制或问题
- 有对后续 feature 有价值的决策经验
```

#### 3.3 Agent 超时优化

**优化方案**：
```
对于 10+ 任务的 feature：
1. 拆分为 2-3 个并行 agent（数据层 / 业务层 / 接口层）
2. 每个 agent 只负责 3-4 个任务
3. 设置合理的超时（15 min per agent）
4. 使用 session_id 续传失败的 agent
```

## 总结

### 做得好的

1. **流程完整性**：init → feature → code → commit 全流程可执行，门禁有效
2. **脚本化程度高**：init-project.js、update-spec-state.js 减少了 AI 手工操作
3. **SPEC-STATE 门禁有效**：格式校验、前置条件检查都能正确拦截
4. **代码质量可接受**：生成的 Java 代码遵循规范，决策合规

### 需要优先改善的

1. **框架安装膨胀**（P0）：178 个文件的提交中 80% 是框架副本，必须重构安装机制
2. **审查环节缺失**（P0）：规划了 compliance-reviewer 但从未执行，要么自动化要么移除
3. **测试文档补作业**（P2）：测试计划和报告应在执行过程中自然产生，而非事后补写
4. **知识沉淀未激活**（P3）：learnings → knowledge 链路全程未触发

### 投入产出评估

| 优化项 | 投入 | 收益 | 优先级 |
|--------|------|------|--------|
| 框架安装重构 | 中（改 install.sh + manifests） | 高（减少 80% 提交膨胀） | P0 |
| 审查自动化 | 中（集成 compliance-reviewer） | 高（填补最大流程缺口） | P0 |
| .gitignore 自动创建 | 低（init-project.js 加一行） | 中（防止构建产物提交） | P1 |
| SPEC-STATE 模板修复 | 低（改模板或脚本） | 中（减少格式调试） | P1 |
| 测试文档前置 | 中（改 /tech:code 流程） | 中（改善文档质量） | P2 |
| 知识沉淀激活 | 低（commit skill 加检查） | 低（当前价值有限） | P3 |
