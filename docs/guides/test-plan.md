# 测试计划指南

本文档面向被 tinypowers 初始化后的目标项目，用来说明如何为一个需求准备测试计划。

它不是单元测试规则的替代品，而是把“这次需求要测什么、怎么测、做到什么程度”说清楚。

## 测试计划要回答什么

一份有效的测试计划，至少应回答：
- 本次测试覆盖哪些范围
- 哪些类型的测试会执行
- 关键用例有哪些
- 覆盖率目标是什么
- 提测前需要产出什么结果

## 测试策略

默认沿用测试金字塔：

```text
E2E 测试         少量关键业务路径
集成测试         API、Service、数据流转
单元测试         核心业务逻辑主力覆盖
```

原则：
- 快速反馈：单元测试优先
- 独立性：测试尽量互不依赖
- 可重复：相同输入得到相同结果
- 自验证：测试自己判断通过或失败

## 常见测试类型

| 类型 | 目标 | 常见范围 |
|------|------|---------|
| 单元测试 | 验证最小可测单元正确性 | Business、工具类、枚举、校验逻辑 |
| 集成测试 | 验证多层协作正确性 | Controller -> Service -> Mapper |
| E2E 测试 | 验证关键用户路径 | 少量关键业务场景 |

## 用例设计建议

### 命名

推荐格式：

```text
test[方法名]_[场景描述]_[预期结果]
```

### 结构

推荐使用 Given / When / Then：

```java
@Test
void testScenario() {
    // Given
    TaskRequest request = createValidRequest();

    // When
    Task result = taskService.createTask(request);

    // Then
    assertNotNull(result.getId());
}
```

### 必测场景

| 类别 | 必测内容 |
|------|---------|
| 正常路径 | 有效输入、成功创建 / 查询 / 更新 / 删除 |
| 边界条件 | 空值、零值、最大值、空字符串 |
| 异常路径 | 找不到记录、无效参数、违反约束 |
| 状态转换 | 核心状态流转 |
| 枚举覆盖 | 每个关键枚举值至少被触达 |

### 参数校验

建议至少覆盖：
- 必填参数为空
- 字符串长度超限
- 格式校验失败

## 测试数据管理

默认建议：
- 测试数据可复用、可清理
- 能回滚时优先回滚
- 构建复杂数据时优先使用 Builder 或工厂方法

## 覆盖率目标

默认目标：

| 指标 | 目标 |
|------|------|
| 行覆盖率 | ≥ 80% |
| 分支覆盖率 | ≥ 70% |
| Business 层覆盖率 | ≥ 90% |

注意：
- 覆盖率是下限，不是目的本身
- 关键业务的异常和边界场景，比“凑覆盖率”更重要

## API 用例清单怎么写

建议对每个关键接口至少列出：
- 正常请求
- 参数错误
- 资源不存在
- 权限不足或状态不允许

示例：

| 用例 | 输入 | 预期结果 |
|------|------|---------|
| 正常创建 | 完整信息 | 201 + 返回实体 |
| 空标题 | `title=\"\"` | 400 + 校验错误 |
| ID 不存在 | `GET /api/tasks/999` | 404 或明确错误响应 |

## 运行测试

```bash
# 所有测试
mvn test

# 单个测试类
mvn test -Dtest=CustomerAssignmentBusinessTest

# 覆盖率报告
mvn test jacoco:report
```

## 提测前建议产物

提测前最好准备：
- 测试范围说明
- 关键用例清单
- 执行结果汇总
- 已知缺陷列表
- 测试结论

## TDD 建议

如果团队采用 TDD，建议遵守：

1. 红：先写失败测试
2. 绿：写最小实现让测试通过
3. 重构：在测试保护下整理实现

## 一个好的测试计划是什么样

读完测试计划后，团队应该能明确：

1. 哪些内容这次一定会测
2. 哪些内容暂时不测或降级测
3. 用什么方式验证关键场景
4. 什么结果可以认为具备提测条件
