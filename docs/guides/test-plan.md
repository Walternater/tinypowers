# 测试计划规范

本文档定义 atomic-task 项目的测试策略、测试类型、测试用例编写规范及覆盖率要求。

---

## 一、测试策略

### 1.1 测试金字塔

```
                    ┌─────────────┐
                    │   E2E 测试   │  少量关键场景
                   ─┴─────────────┴─
                  ┌───────────────────┐
                  │   集成测试         │  API、Service层
                 ─┴───────────────────┴─
                ┌─────────────────────────┐
                │      单元测试            │  核心业务逻辑
               ─┴─────────────────────────┴─
```

### 1.2 测试原则

- **快速反馈**：单元测试优先，失败后逐层向外定位
- **独立性**：每个测试用例独立运行，不依赖其他测试结果
- **可重复**：相同输入在任何环境下产生相同结果
- **自验证**：测试用例自己判断结果是否通过

---

## 二、测试类型

| 类型 | 目标 | 范围 | 技术栈 |
|------|------|------|--------|
| 单元测试 | 验证最小可测试单元的正确性 | Business 层业务逻辑、工具类、枚举类 | JUnit 5、Mockito |
| 集成测试 | 验证多层组件协同工作的正确性 | Controller → Service → Mapper 完整调用链 | Spring Boot Test、MockMvc、H2 |
| E2E 测试 | 验证端到端业务流程 | 少量关键场景 | - |

## 三、测试用例编写规范

### 3.1 命名规范

```
test[方法名]_[场景描述]_[预期结果]
```

**示例**：
- `testCreateTask_withValidInput_returnsCreatedTask`
- `testCreateTask_withNullTitle_throwsValidationException`
- `testGetTask_byId_notFound_returnsNull`

### 3.2 测试用例结构（Given-When-Then）

```java
@Test
void testScenario() {
    // ========== Given（准备测试数据）==========
    TaskRequest request = createValidRequest();

    // ========== When（执行被测操作）==========
    Task result = taskService.createTask(request);

    // ========== Then（验证结果）==========
    assertNotNull(result.getId());
}
```

### 3.3 必须覆盖的场景

| 类别 | 必须覆盖的场景 |
|------|---------------|
| 正常路径 | 有效输入、成功创建/查询/更新/删除 |
| 边界条件 | 空值、零值、最大值、空字符串 |
| 异常路径 | 找不到记录、无效参数、违反约束 |
| 状态转换 | 任务状态变更 |
| 枚举覆盖 | 每个枚举值都有对应的测试用例 |

### 3.4 枚举测试规范

每个枚举必须包含：
- `ofType` 方法对有效码值的测试
- `ofType` 方法对无效码值返回 null 的测试
- 所有枚举值的 code 有效性测试

### 3.5 参数校验测试

必须覆盖：
- 必填参数为空
- 字符串长度超限
- 格式校验（邮箱、手机号等）

## 四、测试数据管理

### 4.1 事务回滚

使用 `@Transactional` 确保测试数据自动回滚。

### 4.2 测试数据构建器

使用 Builder 模式构建测试数据，便于复用和定制。

## 六、API 测试用例清单

### 6.1 任务创建

| 用例 | 输入 | 预期结果 |
|------|------|---------|
| 正常创建 | 完整信息 | 201 + 返回任务 |
| 空标题 | title="" | 400 + 校验错误 |

### 6.2 任务查询

| 用例 | 输入 | 预期结果 |
|------|------|---------|
| ID查询存在 | GET /api/tasks/1 | 200 + 任务 |
| ID查询不存在 | GET /api/tasks/999 | 404 或 null |

### 6.3 任务更新

| 用例 | 输入 | 预期结果 |
|------|------|---------|
| 正常更新 | 完整信息 | 200 + 更新后任务 |
| ID不存在 | PUT /api/tasks/999 | 404 或异常 |

## 七、运行测试

```bash
# 运行所有测试
mvn test

# 运行单个测试类
mvn test -Dtest=CustomerAssignmentBusinessTest

# 生成覆盖率报告
mvn test jacoco:report
```

## 八、测试报告

提测前必须按照模板填写测试报告：`configs/templates/test-report.md`

测试报告必须包含：
- 测试基本信息（需求名称、分支、测试环境、测试人员、时间）
- 测试范围（功能点、测试类型）
- 用例执行汇总（通过率）
- 缺陷汇总（严重/高级/中级/低级分布）
- 测试结论（通过/不通过/有条件通过）

## 九、TDD红绿重构流程

### 强制顺序
1. 红：编写失败的测试用例
2. 绿：编写最小化实现通过测试
3. 重构：优化代码结构

### 执行要求
- 测试先行：先写测试，再写实现
- 最小实现：代码只需通过测试，不需要过度设计
- 持续重构：在测试保护下持续优化代码

## 十、结构化测试数据

### JSON格式
```json
{
  "testcase_id": "TC-001",
  "module": "OrderService",
  "method": "calculatePoints",
  "scenarios": [
    {
      "name": "正常消费",
      "input": {"amount": 100},
      "expected": {"points": 100}
    },
    {
      "name": "满减赠送",
      "input": {"amount": 200, "promotion": "double"},
      "expected": {"points": 400}
    }
  ]
}
```

### 字段定义
- testcase_id: 测试用例唯一标识
- module: 所属模块
- method: 被测方法
- scenarios: 测试场景数组
- name: 场景名称
- input: 输入参数
- expected: 预期结果
