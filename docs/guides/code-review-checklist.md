# 代码审查清单

提交代码审查时，对照以下标准进行检查。

---

## 一、严重问题（必须修复）

### 1.1 数据类型一致性

- [ ] **请求类与实体类类型匹配**：请求DTO字段类型与实体类字段类型一致
- [ ] **实体类与数据库类型匹配**：Java类型与数据库字段类型兼容（如Long对应BIGINT）
- [ ] **跨层类型转换无精度丢失**：ID字段统一使用Long，避免Integer转Long精度丢失
- [ ] **时间类型一致**：统一使用`LocalDateTime`，避免Date与LocalDateTime混用

### 1.2 分页查询正确性

- [ ] **offset计算正确**：`LIMIT (pageNum - 1) * pageSize, pageSize`
- [ ] **pageNum从1开始**：第一页pageNum=1，第二页pageNum=2
- [ ] **避免直接使用pageNum作为起始**：计算offset变量并正确传递到SQL
- [ ] **分页参数边界校验**：pageSize不超过最大值（如100）

### 1.3 枚举值完整性

- [ ] **枚举覆盖所有业务场景**：对照需求文档检查所有状态/类型是否都有枚举值
- [ ] **枚举值与需求一致**：枚举描述与业务语义完全匹配
- [ ] **新增枚举向后兼容**：新增枚举值不影响已有逻辑

### 1.4 安全问题

- [ ] **SQL注入防护**：使用`#{}`而非`${}`拼接SQL参数
- [ ] **敏感信息脱敏**：日志中不输出密码、token等敏感数据
- [ ] **权限校验**：接口是否有必要的权限校验

---

## 二、警告问题（强烈建议修复）

### 2.1 性能问题

- [ ] **避免全量加载**：不使用`Integer.MAX_VALUE`加载全量数据到内存统计
- [ ] **批量操作优化**：批量插入/更新使用批量SQL而非循环单条
- [ ] **数据库聚合替代内存计算**：统计类需求使用`COUNT/SUM/AVG`等SQL函数
- [ ] **N+1查询检查**：循环内是否存在数据库查询

### 2.2 参数校验完整性

- [ ] **null检查**：必填参数有null校验
- [ ] **空字符串检查**：`""`和`null`需要区分处理
- [ ] **长度边界校验**：字符串长度限制、集合大小限制
- [ ] **格式校验**：手机号、邮箱、日期等格式校验

```java
// 正确示例
if (req.getFeedbackStatus() == 2) {
    if (req.getFeedbackContent() == null || req.getFeedbackContent().trim().isEmpty()) {
        throw new IllegalArgumentException("反馈内容不能为空");
    }
    if (req.getFeedbackContent().length() > MAX_LENGTH) {
        throw new IllegalArgumentException("反馈内容不能超过" + MAX_LENGTH + "字");
    }
}
```

### 2.3 防重复与幂等性

- [ ] **唯一索引**：关键业务字段是否有唯一索引
- [ ] **插入前检查**：新增操作前检查是否已存在重复记录
- [ ] **幂等性设计**：重复调用是否会产生副作用

### 2.4 枚举合法性校验

- [ ] **ofType返回null检查**：使用枚举前检查`ofType`返回值
- [ ] **枚举值范围校验**：入参是否为有效枚举值

```java
// 正确示例
CarSaleCertaintyEnum certainty = CarSaleCertaintyEnum.ofType(req.getCarSaleCertainty());
if (certainty == null) {
    throw new IllegalArgumentException("车辆售卖确定性枚举值无效");
}
```

### 2.5 异常处理

- [ ] **异常信息明确**：抛出异常时提供有意义的错误信息
- [ ] **异常不吞掉**：catch块中有实际处理或日志记录
- [ ] **事务回滚**：写操作异常确保事务正确回滚

---

## 三、建议问题（可选优化）

### 3.1 代码风格

- [ ] **注释风格统一**：`/** */`文档注释与`//`单行注释不混用
- [ ] **命名规范**：变量/方法/类名符合驼峰命名或项目既定规范
- [ ] **类文件结构**：字段→构造方法→公共方法→私有方法排列有序

### 3.2 常量与魔法数字

- [ ] **硬编码值抽取**：时间格式、超时值等硬编码值抽取为常量
- [ ] **魔法数字替代**：状态码、类型码使用枚举或常量类
- [ ] **字符串字面量**：重复使用的字符串抽取为常量

```java
// 错误
DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss");
if (status == 1) { ... }

// 正确
private static final DateTimeFormatter DATE_TIME_FORMATTER = DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss");
if (status == TaskStatus.PENDING.getCode()) { ... }
```

### 3.3 日志规范

- [ ] **日志级别正确**：INFO记录业务流程，WARN记录可恢复问题，ERROR记录异常
- [ ] **日志包含上下文**：`log.info("Task created: id={}", task.getId())`
- [ ] **敏感信息屏蔽**：用户密码、token等不写入日志

### 3.4 代码冗余

- [ ] **未使用的代码**：删除未调用的方法、变量
- [ ] **重复代码抽取**：相似逻辑抽取为公共方法
- [ ] **Mapper方法利用**：已定义的Mapper方法是否充分利用

## 五、问题修复优先级

### 第一优先级（阻塞）
1. 数据类型不一致
2. 分页查询实现错误
3. 枚举值缺失
4. SQL注入风险

### 第二优先级（强烈建议）
1. 性能问题（全量加载）
2. 参数校验逻辑缺陷
3. 缺少防重复校验
4. 枚举值合法性校验缺失

### 第三优先级（可选优化）
1. 代码注释风格
2. 硬编码值抽取
3. 魔法数字替代
4. 日志规范

---

## 六、多Agent审查流程

### 审查阶段
1. 安全审查Agent → 输出安全报告
2. 代码审查Agent → 输出质量报告
3. 性能审查Agent → 输出性能报告
4. 人工终审 → 最终决策

### 审查维度

#### 安全性
- SQL注入：确认所有查询使用参数化
- XSS：输入输出是否经过滤
- 权限绕过：是否有@PreAuthorize等权限校验
- 敏感数据：日志、输出是否脱敏

#### 可维护性
- 圈复杂度：方法不超过15
- 重复代码：相似逻辑是否抽取
- 耦合度：模块间依赖是否清晰

#### 性能
- N+1查询：循环内是否有数据库查询
- 批量处理：是否使用批量SQL
- 缓存策略：热点数据是否缓存
