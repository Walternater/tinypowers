# PRD 分析指南

本文档面向使用 tinypowers 的开发者，说明如何把一个 PRD 转成可执行的需求定义。

目标不是“读完 PRD 就开始写代码”，而是先回答清楚三件事：
- 到底要做什么
- 为什么要这样做
- 交付时怎么判断算完成

## 这份文档适合什么时候看

主要用于 `/tech:feature` 前半段，尤其是：
- 刚拿到 PRD 时
- 需求有歧义时
- 需要拆成 Story / Task 时

## PRD 分析目标

一个合格的分析结果，至少应覆盖：
- 需求背景
- 目标用户
- 功能范围
- 验收标准
- 非功能需求
- 待澄清问题

## 推荐分析路径

```text
收到 PRD
  -> 提炼背景和目标
  -> 识别功能范围
  -> 提炼验收标准
  -> 识别歧义和边界
  -> 形成需求理解确认
  -> 进入技术方案和任务拆解
```

## 第一部分：需求理解

拿到 PRD 后，优先确认：

- [ ] 为什么做这个需求
- [ ] 谁会使用这个功能
- [ ] 哪些是本次必须交付
- [ ] 哪些是本次明确不做
- [ ] 验收时看哪些指标或行为
- [ ] 是否有性能、安全、兼容性要求

## 第二部分：歧义检测

技术方案前，优先找出这些模糊点：

- 需求描述是否有模糊词
- 边界条件是否明确
- 异常场景是否覆盖
- 数据量级是否清晰
- 性能要求是否量化

推荐输出格式：

```markdown
## 待澄清问题

| 序号 | 模糊点 | 影响 | 优先级 |
|------|--------|------|--------|
| 1 | ... | ... | 高/中/低 |
```

## 第三部分：需求理解确认

推荐形成一个简洁确认稿，至少包含：

```text
需求名称：【CSS-xxx】XXX 功能
需求链接：JIRA/Wiki 地址

一、我对需求的理解
1. 背景：...
2. 目标：...
3. 核心功能点：...

二、我的疑问
1. ...
2. ...

三、技术建议（如有）
1. ...
```

## 第四部分：任务拆解思路

### 拆解原则

| 原则 | 说明 |
|------|------|
| 独立可开发 | 单个任务尽量能独立推进 |
| 独立可测试 | 单个任务应能明确验证 |
| 粒度适中 | 避免任务过大或过碎 |
| 接口先行 | 有前后端协作时优先明确接口 |

### 常见拆解维度

后端常见任务：
- 数据库设计
- 实体 / DTO
- 枚举定义
- Mapper / DAO
- Business
- Service
- Controller
- 单元测试

前端常见任务：
- 页面结构
- 接口对接
- 表单处理
- 列表展示
- 状态管理
- 路由配置
- 样式与交互
- 组件测试

### 任务表应至少包含

| 字段 | 说明 |
|------|------|
| 任务编号 | 如 `T-001` |
| 任务名称 | 简洁明确 |
| 类型 | 后端 / 前端 / DB / 测试 |
| 预估工时 | 可用于 Wave 规划 |
| 依赖任务 | 明确前置关系 |
| 验收标准 | 可验证，不模糊 |

## 第五部分：技术方案准备

进入技术方案前，至少确认：

- [ ] 分层与职责边界清晰
- [ ] 接口方向明确
- [ ] 数据模型已有初步思路
- [ ] 关键风险已被识别
- [ ] 有需要锁定的关键决策

## 第六部分：估算建议

可用的粗粒度方法：

| 方法 | 适用场景 |
|------|---------|
| 类比估算 | 有历史相似需求 |
| 分解估算 | 任务能进一步拆分 |
| 三点估算 | 不确定性较高 |

经验建议：
- 新技术或外部依赖场景应加风险系数
- 复杂需求不要把测试成本漏掉
- 如果一个任务已经大到难以估算，通常说明还该继续拆

## 一个好的分析结果应该是什么样

读完分析结果后，团队应该能回答：

1. 做这个需求的业务原因是什么？
2. 本次范围和非范围分别是什么？
3. 哪些点还没澄清？
4. 技术方案需要重点解决什么？
5. 后续任务应该先拆哪里？

如果这些问题还答不出来，说明 PRD 分析还不够完整。
