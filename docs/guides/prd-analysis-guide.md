# 产品需求 PRD 分析与任务拆解指南

本文档提供产品需求文档（PRD）的分析方法、前后端任务拆解规范。

---

## 一、PRD 分析流程

```
收到 PRD → 需求理解 → 技术方案设计 → 任务拆解 → 技术评审 → 开发执行
```

### 1.1 需求理解检查

收到 PRD 后，24 小时内完成以下理解确认：

- [ ] **需求背景**：为什么做这个功能？解决什么问题？
- [ ] **目标用户**：谁会使用这个功能？他们的使用场景是什么？
- [ ] **功能范围**：哪些是必须做的？哪些是可选的？
- [ ] **验收标准**：怎么算功能完成？有哪些明确的指标？
- [ ] **非功能需求**：性能要求、安全要求、兼容性要求？

### 1.2 歧义检测环节

### 执行时机
需求分析后，技术方案前

### 执行内容
1. 识别需求中的模糊点
2. 生成待澄清问题列表
3. 输出 CSS-2005-questions.md

### 问题模板
```markdown
## 待澄清问题

| 序号 | 模糊点 | 影响 | 优先级 |
|------|--------|------|--------|
| 1    | ...    | ...  | 高/中/低 |
```

### 歧义检测检查项
- [ ] 需求描述是否有歧义？
- [ ] 边界条件是否明确？
- [ ] 异常场景是否覆盖？
- [ ] 数据量级是否清晰？
- [ ] 性能要求是否量化？

### 1.3 需求理解确认模板

```
需求名称：【CSS-xxx】XXX 功能
需求链接：JIRA/Wiki 地址

一、我对需求的理解
1. 背景：...
2. 目标：...
3. 核心功能点：
   - 功能点 A：...
   - 功能点 B：...
   - 功能点 C：...

二、我的疑问
1. 问题 1：...
2. 问题 2：...

三、技术建议（如有）
1. 建议 1：...
```

## 二、功能拆解

### 2.1 功能拆解原则

| 原则 | 说明 |
|------|------|
| **独立可开发** | 每个任务可以独立开发，不依赖其他任务 |
| **独立可测试** | 每个任务可以独立测试，验证功能正确性 |
| **粒度适中** | 一个任务 1-3 人天完成，避免过大任务 |
| **接口先行** | 前后端接口先定义，后端先行开发 |

### 2.2 功能拆解示例

**原始需求**：客户分配任务功能

```
├── 前端
│   ├── 客户列表页
│   │   ├── 列表展示
│   │   ├── 筛选功能
│   │   └── 分页功能
│   ├── 客户分配页
│   │   ├── 销售选择
│   │   └── 确认分配
│   └── 分配记录页
│
└── 后端
    ├── 客户分配 API
    │   ├── POST /api/customer/assign（分配）
    │   ├── GET /api/customer/assign/list（分配记录）
    │   └── ...
    │
    └── 业务逻辑
        ├── 分配规则校验
        ├── 重复分配检查
        └── 分配记录生成
```

## 六、任务估算

### 6.1 估算方法

| 方法 | 适用场景 |
|------|---------|
| **类比估算** | 有类似经验，参考历史任务 |
| **分解估算** | 大任务拆分为小任务分别估算 |
| **三点估算** | (乐观+4×最可能+悲观)/6 |

### 6.2 估算校准

| 原始估算 | 风险系数 | 实际估算 |
|---------|---------|---------|
| 1d | × 1.2 | 1.2d |
| 2d | × 1.3 | 2.6d |
| 3d+ | × 1.4 | 4.2d+ |

**风险场景**：
- 新技术/框架
- 复杂业务逻辑
- 外部系统依赖
- 不确定性高

## 七、PRD 分析案例

### 7.1 原始 PRD 示例

```
【CSS-xxx】客户分配优先级功能

一、需求背景
用户需要为客户分配设置优先级，以便区分重要程度，合理安排工作。

二、功能需求
1. 分配时可选择优先级（高/中/低）
2. 客户列表按优先级排序（高 > 中 > 低）
3. 支持按优先级筛选客户
4. 优先级变更有记录

三、验收标准
1. 分配可设置优先级
2. 列表默认按优先级排序
3. 筛选器可按优先级筛选
4. 优先级变更记录可查看

四、非功能需求
1. 支持 10 万级客户数据
2. 接口响应时间 < 200ms
```

### 7.2 需求理解确认

```
一、需求理解
1. 背景：用户需要优先级功能区分客户重要程度
2. 目标：支持优先级设置、排序、筛选
3. 核心功能点：
   - 优先级枚举：高(HIGH=1)、中(MEDIUM=2)、低(LOW=3)
   - 分配时设置优先级
   - 列表按优先级排序
   - 筛选器支持优先级筛选
   - 优先级变更记录表

二、疑问
1. 优先级变更记录保留多久？
2. 列表是否支持手动调整优先级？
3. 优先级是否有数量限制？

三、技术建议
1. 复用现有的 CustomerAssignment 实体，新增 priority 字段
2. 新增优先级变更记录表 customer_priority_change_log
3. Mapper 新增排序查询方法
```

### 7.3 任务拆解

**后端任务**：

| 任务 | 预估 | 依赖 |
|------|------|------|
| T-001：CustomerAssignment 实体新增 priority 字段 | 0.5d | - |
| T-002：PriorityEnum 枚举完善 | 0.5d | - |
| T-003：分配接口支持优先级 | 0.5d | T-001, T-002 |
| T-004：客户列表接口支持优先级排序 | 1d | T-001 |
| T-005：客户列表接口支持优先级筛选 | 0.5d | T-004 |
| T-006：优先级变更记录表设计 | 0.5d | - |
| T-007：优先级变更记录接口 | 1d | T-006 |
| T-008：后端单元测试 | 1d | T-003~T-007 |
