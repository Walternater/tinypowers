# Knowledge Capture 规则设计

本文档定义 `/tech:commit` 阶段知识提取的完整规则，包括触发条件、写入格式、去重逻辑和来源追溯。

---

## 1. 四类知识触发条件

### 1.1 约定 (Convention)

**定义**: 项目特有的编码约定、命名规范、结构约定等非标准实践。

**触发条件**:
| 场景 | 示例 | 是否捕获 |
|------|------|----------|
| 命名风格与常规不同 | 使用 `findXxx` 而非 `getXxx` 作为查询方法前缀 | ✅ 捕获 |
| 目录结构有特殊含义 | `config/` 目录下的文件自动加载 | ✅ 捕获 |
| 注释有特殊格式 | 使用 `// FIXME(username):` 标记 | ✅ 捕获 |
| 通用标准实践 | 使用驼峰命名法 | ❌ 不捕获 |

**捕获模板**:
```markdown
### 约定: [简短描述]

**主题**: [命名/结构/注释/其他]
**约束**: [具体约定内容]
**场景**: [适用场景]
**来源**: Feature [FEAT-XXX] - [简要说明]
**日期**: YYYY-MM-DD
```

---

### 1.2 踩坑 (Gotcha)

**定义**: 开发过程中遇到的问题及其解决方案，尤其是非显而易见的陷阱。

**触发条件**:
| 场景 | 示例 | 是否捕获 |
|------|------|----------|
| 调试时间 > 30 分钟的问题 | Spring Boot 测试上下文缓存导致的配置不生效 | ✅ 捕获 |
| 反直觉的框架行为 | JPA `save()` 后实体状态变更 | ✅ 捕获 |
| 环境相关的问题 | 特定 JDK 版本的兼容性问题 | ✅ 捕获 |
| 已记录在官方文档的预期行为 | JPA 懒加载异常 | ❌ 不捕获 |

**捕获模板**:
```markdown
### 踩坑: [问题简述]

**问题**: [详细描述]
**原因**: [根本原因分析]
**解决**: [解决方案]
**预防**: [如何避免再次发生]
**来源**: Feature [FEAT-XXX] - [Task ID]
**日期**: YYYY-MM-DD
```

---

### 1.3 模式 (Pattern)

**定义**: 项目中反复出现的代码实现模式或设计模式。

**触发条件**:
| 场景 | 示例 | 是否捕获 |
|------|------|----------|
| 出现 3+ 次的相似实现 | 统一的异常处理包装模式 | ✅ 捕获 |
| 跨项目的通用模式 | 领域事件发布/订阅模式 | ✅ 捕获 |
| 特定场景的标准做法 | 批量操作的统一处理方式 | ✅ 捕获 |
| 只出现 1-2 次的实现 | 特定的工具方法 | ❌ 不捕获 |

**捕获模板**:
```markdown
### 模式: [模式名称]

**描述**: [模式用途简述]
**适用场景**: [何时使用]
**示例**:
```java
[代码示例]
```
**变体**: [如有多种实现方式]
**来源**: Feature [FEAT-XXX]
**日期**: YYYY-MM-DD
```

---

### 1.4 重构 (Refactoring)

**定义**: 代码重构过程中的关键决策、教训和最佳实践。

**触发条件**:
| 场景 | 示例 | 是否捕获 |
|------|------|----------|
| 重大结构变更 | 从贫血模型重构为充血模型 | ✅ 捕获 |
| 性能优化经验 | 查询优化从 N+1 到批量加载 | ✅ 捕获 |
| 技术债务清理 | 统一异常处理重构 | ✅ 捕获 |
| 简单的代码整理 | 变量重命名 | ❌ 不捕获 |

**捕获模板**:
```markdown
### 重构: [重构主题]

**背景**: [为什么需要重构]
**变更**: [具体做了什么]
**挑战**: [遇到的问题]
**收获**: [关键教训]
**迁移指南**: [如何应用到其他模块]
**来源**: Feature [FEAT-XXX]
**日期**: YYYY-MM-DD
```

## 2. 知识写入格式

### 2.1 写入位置

知识写入 `docs/knowledge.md`，按类别分章节：

```markdown
# 领域知识

## 约定

[按时间倒序排列的约定条目]

## 踩坑

[按时间倒序排列的踩坑条目]

## 模式

[按时间倒序排列的模式条目]

## 重构

[按时间倒序排列的重构条目]
```

### 2.2 格式规范

1. **标题级别**: 使用 `###` (三级标题)
2. **排序规则**: 按日期倒序，新知识在顶部
3. **字段完整性**: 模板中所有字段必须填写
4. **代码块**: 示例代码使用 fenced code blocks 并标注语言

## 3. 知识去重逻辑

### 3.1 去重策略

| 知识类型 | 去重字段 | 相似度阈值 | 处理方式 |
|----------|----------|------------|----------|
| 约定 | 主题 + 约束内容 | 80% | 合并或更新 |
| 踩坑 | 问题描述 | 70% | 引用已有条目 |
| 模式 | 模式名称 + 适用场景 | 75% | 添加变体说明 |
| 重构 | 变更内容 | 70% | 更新原条目 |

### 3.2 去重检查流程

```
发现新知识
    │
    ▼
┌─────────────────┐
│ 提取关键词      │
│ (主题/场景/技术) │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 搜索现有知识    │
│ grep 关键词     │
└────────┬────────┘
         │
    ┌────┴────┐
    ▼         ▼
  相似      不相似
    │         │
    ▼         ▼
┌────────┐  ┌────────┐
│ 评估   │  │ 写入新 │
│ 合并   │  │ 条目   │
│ 或跳过 │  │        │
└────────┘  └────────┘
```

### 3.3 去重判断示例

**场景**: 发现新约定 "Controller 方法使用 `listXxx` 前缀返回列表"

1. **提取关键词**: `Controller`, `list`, `前缀`, `命名`
2. **搜索**: `grep -i "controller\|前缀\|命名" docs/knowledge.md`
3. **发现相似**:
   ```markdown
   ### 约定: Service 查询方法使用 findXxx 前缀
   **主题**: 命名
   **约束**: 查询方法统一使用 find 前缀而非 get
   ```
4. **评估**: 同属命名约定，可以合并为通用约定
5. **处理**: 更新原条目，扩展适用范围

## 4. 知识来源追溯

### 4.1 追溯字段

每条知识必须包含完整的来源信息：

| 字段 | 格式 | 示例 |
|------|------|------|
| Feature | `Feature [项目前缀-数字]` | Feature TINYPOWERS-042 |
| Task | `Task [T-XXX]` | Task T-003 |
| Commit | `Commit [hash]` | Commit a1b2c3d |
| 日期 | `YYYY-MM-DD` | 2026-04-09 |

### 4.2 追溯用途

1. **问题追踪**: 当约定变更时，可以追溯到原始需求
2. **影响分析**: 了解知识条目的背景和上下文
3. **知识演化**: 追踪同一主题的多条知识如何演进

### 4.3 追溯查询

```bash
# 查找特定 Feature 相关的知识
grep -A 10 "Feature TINYPOWERS-042" docs/knowledge.md

# 按日期范围查找知识
grep -B 2 "2026-04" docs/knowledge.md

# 查找某人的贡献
grep -B 5 "作者: username" docs/knowledge.md
```

```
┌──────────────────────────────────────────────┐
│ Commit Phase 开始                             │
└──────────────┬───────────────────────────────┘
               │
               ▼
┌──────────────────────────────────────────────┐
│ 1. 扫描开发产出                                │
│    - 代码变更                                  │
│    - 解决的问题                                │
│    - 实现的方案                                │
└──────────────┬───────────────────────────────┘
               │
               ▼
┌──────────────────────────────────────────────┐
│ 2. 识别知识类型                                │
│    - 是否符合约定触发条件?                      │
│    - 是否符合踩坑触发条件?                      │
│    - 是否符合模式触发条件?                      │
│    - 是否符合重构触发条件?                      │
└──────────────┬───────────────────────────────┘
               │
               ▼
┌──────────────────────────────────────────────┐
│ 3. 去重检查                                    │
│    - 提取关键词                                │
│    - 搜索现有知识                              │
│    - 评估相似度                                │
└──────────────┬───────────────────────────────┘
               │
        ┌──────┴──────┐
        ▼             ▼
   已存在          新知识
        │             │
        ▼             ▼
   合并/跳过     按模板格式化
                        │
                        ▼
           ┌─────────────────────────┐
           │ 4. 写入 knowledge.md    │
           │    - 选择正确章节       │
           │    - 按时间倒序插入     │
           │    - 更新目录索引       │
           └─────────────────────────┘
```

## 6. 质量检查清单

知识捕获完成后，检查以下项目：

- [ ] 所有新知识已分类 (约定/踩坑/模式/重构)
- [ ] 所有条目包含完整的来源追溯信息
- [ ] 无重复知识 (或已正确合并)
- [ ] 代码示例可运行或语法正确
- [ ] 日期格式统一为 YYYY-MM-DD
- [ ] Feature 编号格式正确
