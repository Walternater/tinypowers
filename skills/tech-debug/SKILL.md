# /tech:debug

## 作用

`/tech:debug` 用于系统化地排查和修复问题。

它回答四个问题：
- 问题现象是什么
- 可能的根因有哪些
- 哪个假设是正确的
- 修复后是否真正解决

## 输入

- 错误信息、异常日志或复现步骤
- 相关代码文件路径（如有）
- `features/{id}/STATE.md`（如果在需求流程中遇到问题）

## 默认 Context

默认进入 `contexts/debug.md`。

如果只是收集信息、不打算修改代码，可以暂时切到 `contexts/research.md`；
确认根因后再回到 `contexts/debug.md` 完成验证和修复。

## 主流程

```text
Observe → Hypothesize → Verify → Fix → Validate
```

这条链路必须顺序推进，每次只验证一个假设。

## 硬约束

- 禁止猜测：先收集信息再假设
- 禁止同时改多处：一次只改一个变量
- 禁止跳过验证：修了不等于好了
- 连续 3 次失败后停止，上升到架构层讨论
- 禁止在根因确认前修改代码

## Phase 1: Observe（观察）

目标是完整收集错误信息。

### 必做事项

- 记录错误信息原文
- 记录复现步骤
- 确认影响范围（哪些接口、哪些数据）
- 查看相关日志

### 输出

```markdown
## 观察

**错误信息:** {原文}
**复现步骤:** {步骤}
**影响范围:** {描述}
**环境:** {分支/版本/环境}
```

## Phase 2: Hypothesize（假设）

基于观察结果提出候选根因。

### 规则

- 提出 1-3 个假设，按可能性排序
- 每个假设必须可验证（有明确的验证方式）
- 不确定的假设标注置信度

### 输出

```markdown
## 假设

| # | 假设 | 置信度 | 验证方式 |
|---|------|--------|----------|
| 1 | {最可能的原因} | 高 | {如何验证} |
| 2 | {次可能的原因} | 中 | {如何验证} |
```

## Phase 3: Verify（验证）

用最小实验逐个验证假设。

### 规则

- 从最可能的假设开始
- 每次只验证一个假设
- 记录验证过程和结果
- 确认根因后立即停止验证

### 输出

```markdown
## 验证

| # | 假设 | 实验 | 结果 | 结论 |
|---|------|------|------|------|
| 1 | ... | ... | 确认/排除 | ... |

**根因:** {确认的根因}
```

## Phase 4: Fix（修复）

针对确认的根因做最小修复。

### 规则

- 只改根因相关的代码
- 不做"顺便优化"
- 修复要有明确注释说明为什么这样改

## Phase 5: Validate（验证修复）

确认修复有效且无副作用。

### 必做事项

- 复现原始问题，确认已修复
- 运行相关测试
- 检查是否有副作用

### 输出

```markdown
## 验证结果

- 原始问题: [已修复 / 未修复]
- 测试: [通过 / 失败]
- 副作用: [无 / 描述]
```

## 失败处理

- 修复后问题仍存在：回到 Phase 2，提出新假设
- 连续 3 次假设被排除：停止，输出分析报告，建议上升到架构层
- 发现问题比预想的大：暂停，告知用户重新评估

## 与其他 Skill 的关系

- `/tech:debug` 可以在任何阶段触发
- 如果在 `/tech:code` 的 Wave 执行中遇到问题，可以先 `/tech:debug` 再回来
- 调试完成后产出可回到 `/tech:code` 继续审查

## 调试日志

如果问题复杂，建议在 `features/{id}/debug-log.md` 中记录完整过程：

```text
features/{id}/debug-log.md
├── 现象描述
├── 假设列表
├── 验证过程
├── 根因结论
├── 修复方案
└── 验证结果
```
