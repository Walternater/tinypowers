# Deviation Log

记录执行过程中发现的偏差和自动修复。

## 与 deviation-handling.md 的关系

- **deviation-handling.md**：偏差处理的理论框架和升级路径
- **deviation-log.md**：实际发生的偏差的实际记录

本文档是 deviation-handling.md 的配套日志，记录执行过程中按 Deviation Rules 自动处理的所有偏差。

## 格式规范

### 文件位置

```
features/{id}/notepads/deviation-log.md
```

### 头部元信息

```markdown
# Deviation Log: {feature_id}

> 创建时间: YYYY-MM-DD HH:MM | 最后更新: YYYY-MM-DD HH:MM

## 统计摘要

| 指标 | 值 |
|------|-----|
| Rule 1 (Bug修复) | N 次 |
| Rule 2 (功能补全) | N 次 |
| Rule 3 (阻塞解决) | N 次 |
| Rule 4 (架构升级) | N 次 |
| 当前 Wave | Wave N |
| 本 Wave 自动修复次数 | N / 3 |
```

### 记录格式

```markdown
## [YYYY-MM-DD HH:MM] Wave {N}

| 时间 | 任务ID | 规则 | 描述 | 证据 | 处理结果 |
|------|--------|------|------|------|----------|
| HH:MM | T-XXX | Rule 1 | 描述 | 文件:行号 | 已修复 |
```

### 字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| 时间 | 是 | 24小时制，精确到分钟 |
| 任务ID | 是 | T-XXX 格式 |
| 规则 | 是 | Rule 1-4 |
| 描述 | 是 | 简洁描述偏差内容 |
| 证据 | 是 | 文件路径和行号，或命令输出 |
| 处理结果 | 是 | 已修复 / 已升级 / 待确认 |

## 规则说明

| 规则 | 条件 | 动作 | 阈值 |
|------|------|------|------|
| Rule 1 | 发现 Bug | 自动修复，不询问 | 同一 Wave ≤3 次 |
| Rule 2 | 遗漏的关键功能（技术方案明确列出） | 自动补全，不询问 | 同一 Wave ≤3 次 |
| Rule 3 | 遇到阻塞问题 | 自动解决阻塞，不询问 | 同一 Wave ≤3 次 |
| Rule 4 | 架构变更 | 暂停，询问用户 | 无上限 |

## 升级路径

当以下情况发生时，必须升级到 Rule 4：

1. **阈值触发**：同一 Wave 内 Rule 1-3 自动修复总计达到 3 次
2. **主观判断**：Rule 2 的"关键功能"无法对应到技术方案中的明确条目号
3. **架构影响**：修复涉及业务逻辑变更、数据结构变更、API 契约变更
4. **连锁风险**：修复可能影响其他 Wave 的任务

### 升级模板

```markdown
## [YYYY-MM-DD HH:MM] 升级到 Rule 4

**升级原因**：
- [具体原因]

**当前状态**：
- Wave: {N}
- 任务: T-XXX
- 本 Wave 自动修复次数: 3/3

**待确认事项**：
- [需要用户决策的问题 1]
- [需要用户决策的问题 2]

**已收集的信息**：
- [可用的上下文和证据]

**等待用户输入后继续**
```

## 示例

### 示例 1：Bug 修复 (Rule 1)

```markdown
## [2026-03-30 10:30] Wave 2

| 时间 | 任务ID | 规则 | 描述 | 证据 | 处理结果 |
|------|--------|------|------|------|----------|
| 10:30 | T-042 | Rule 1 | 修复空指针异常 | src/main/java/com/example/UserService.java:42 | 已修复 |
```

### 示例 2：功能补全 (Rule 2)

```markdown
## [2026-03-30 10:35] Wave 2

| 时间 | 任务ID | 规则 | 描述 | 证据 | 处理结果 |
|------|--------|------|------|------|----------|
| 10:35 | T-042 | Rule 2 | 补全缺失的参数校验 | 技术方案 D-03 明确要求 | 已修复 |
```

### 示例 3：升级到 Rule 4

```markdown
## [2026-03-30 11:00] 升级到 Rule 4

**升级原因**：
- 同一 Wave 内自动修复已达 3 次阈值
- 本次修复涉及订单状态机逻辑变更

**当前状态**：
- Wave: 2
- 任务: T-045
- 本 Wave 自动修复次数: 3/3

**待确认事项**：
- 订单取消后是否允许重新开启？
- 需要同步更新库存系统吗？

**已收集的信息**：
- 技术方案 D-05 定义了状态转换规则
- 库存系统接口在 src/main/java/com/example/InventoryService.java:28

**等待用户输入后继续**
```

## 与 STATE.md 的同步

每次记录偏差后，应同步更新 STATE.md：

```markdown
## 偏差

- [2026-03-30 10:30] Wave 2: T-042 Rule 1 - 空指针异常 (已修复)
- [2026-03-30 10:35] Wave 2: T-042 Rule 2 - 参数校验补全 (已修复)
- [2026-03-30 11:00] Wave 2: T-045 升级 Rule 4 - 状态机逻辑变更 (待确认)
```

## 审查清单

每次 Wave 完成后检查：

- [ ] 所有偏差都有记录
- [ ] 每条记录包含完整证据
- [ ] 阈值达到时已正确升级
- [ ] 已同步偏差到 STATE.md
- [ ] 升级到 Rule 4 的项目已等待用户输入
