# Model Tiering 指南

## 概述

Model Tiering 是一种根据任务复杂度选择最优模型的成本优化策略。不同任务需要不同能力水平的模型，选择合适的模型可以在保证质量的同时降低成本。

## 层级定义

### Tier 1: 快速/便宜模型

**适用场景**：
- 机械性任务（1-2 文件，明确需求）
- 简单格式化/重构
- 批量操作的简单部分
- 快速探索/原型

**特征**：
- Token 成本低
- 响应速度快
- 适合明确、简单、无歧义的任务

**示例任务**：
- 重命名变量（跨多个文件）
- 添加简单的 getter/setter
- 简单的 import 整理
- 格式化为特定 style

### Tier 2: 标准模型

**适用场景**：
- 集成任务（多文件，涉及接口对接）
- 需要一定推理的逻辑实现
- 需要理解上下文的修改
- 中等复杂度的新功能实现

**特征**：
- 平衡成本和能力
- 适合大多数开发任务
- 可以处理一定复杂度

**示例任务**：
- 实现一个 Service 类的业务逻辑
- 修改多个相关文件的接口
- 编写有业务逻辑的测试
- 重构一个模块的内部结构

### Tier 3: 最强模型

**适用场景**：
- 架构/设计/审查
- 复杂逻辑和决策
- 新技术方案设计
- 安全/性能关键代码审查
- 需要深度推理的问题

**特征**：
- 最高 Token 成本
- 最强推理能力
- 适合关键决策任务

**示例任务**：
- 设计系统架构
- 审查安全敏感代码
- 解决复杂的 bug（需要深度分析）
- 制定技术方案
- 关键代码的代码审查

## 任务复杂度判断

### 判断矩阵

| 维度 | Tier 1 | Tier 2 | Tier 3 |
|------|--------|--------|--------|
| 文件数量 | 1-2 | 3-10 | 10+ |
| 依赖关系 | 无 | 中等 | 复杂 |
| 决策点 | 0-1 | 2-5 | 5+ |
| 业务复杂度 | 简单 | 中等 | 高 |
| 风险级别 | 低 | 中 | 高 |
| 新知识需求 | 无 | 部分 | 大量 |

### 简化判断

**Tier 1 信号**：
- 需求非常明确，无歧义
- 涉及 1-2 个文件
- 不需要理解复杂业务逻辑
- 不涉及跨模块协调

**Tier 2 信号**：
- 需求有一定复杂度
- 涉及 3-10 个文件
- 需要理解业务上下文
- 有一定的接口设计

**Tier 3 信号**：
- 架构决策级别
- 涉及系统级设计
- 安全/性能关键
- 需要深度推理和多轮思考

## 在任务执行中的应用

### 任务描述中的 Model 推荐

在任务拆解表中，每个任务应标注推荐模型：

```markdown
## T-001
### 任务
实现用户认证 API

### 推荐模型
**Tier 2** - 涉及接口设计和业务逻辑

### 验收标准
- [ ] API 符合 REST 规范
- [ ] 错误处理完善
- [ ] 测试覆盖率 >= 80%
```

### Wave 执行中的模型分配

一个 Wave 内的任务可以分配不同模型：
- 无依赖的任务并行时，可分配不同 Tier
- 高 Tier 任务优先执行（关键路径）
- 低 Tier 任务可用更快模型加速

### 模型切换策略

**不切换**：
- 单个任务中途切换模型会丢失上下文
- 保持一致性能更好的结果

**切换**：
- 任务完成后，用更强模型审查结果
- 关键决策点用强模型复核

## 成本优化策略

### 1. 批量 Tier 1

将多个 Tier 1 任务打包执行：
```
T-001 (Tier 1) + T-002 (Tier 1) + T-003 (Tier 1) → 一个 Prompt 一次调用
```

### 2. Pipeline 模式

```
Tier 3 (设计) → Tier 2 (实现) → Tier 1 (完善)
```

### 3. 预估成本

执行前预估 Token 消耗：
```markdown
## T-XXX 预估
- 输入: ~2000 tokens
- 输出: ~1500 tokens
- 总计: ~3500 tokens
- 成本: $0.XXX (Tier X)
```

### 4. 质量保障

**不要为了省钱降级模型导致**：
- 需要重做（成本更高）
- 引入 bug（风险更高）
- 审查失败（浪费时间）

## 场景示例

### 场景 1: 简单重构

**任务**：将 `userName` 重命名为 `username`

**判断**：Tier 1（机械性、明确、无歧义）

**执行**：使用快速模型，批量处理

### 场景 2: 实现业务逻辑

**任务**：实现订单状态机

**判断**：Tier 2（需要理解业务逻辑，有一定复杂度）

**执行**：使用标准模型，设计好接口后实现

### 场景 3: 安全审查

**任务**：审查认证模块安全性

**判断**：Tier 3（安全关键，需要深度分析）

**执行**：使用最强模型，进行全面 OWASP 分析

## HARD-GATE 约束

<HARD-GATE>
**模型选择规则**：
1. 高风险任务（安全、性能、架构）禁止降级到 Tier 1
2. 如果对模型选择不确定，应升级而非降级
3. 关键路径任务必须使用 Tier 2 或以上
</HARD-GATE>
