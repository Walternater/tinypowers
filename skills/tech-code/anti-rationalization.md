# anti-rationalization.md

## 反合理化设计模式

借鉴 superpowers 项目，基于劝说心理学研究（N=28,000 对话，合规率 33% → 72%），
在每个 HARD-GATE 附近附加合理化表格，帮助 Agent 抵抗跳过检查的冲动。

---

## 设计理念

AI Agent 在压力下会「合理化」跳过质量检查的行为。这不是 bug，而是 LLM 的特性。
通过在每个门禁旁明确列出「合理化想法 → 实际风险」的映射，可以在 Agent 的思维链中
提前注入对抗信号。

---

## 通用合理化表格

适用于所有 HARD-GATE：

| 你可能在想 | 实际情况 |
|-----------|----------|
| "这只是简单任务，不需要检查" | 简单任务也需要正确。跳过检查本身就是风险 |
| "我已经确认过了，不需要验证" | 自我确认 ≠ 独立验证。用 task 工具让 Subagent 检查 |
| "用户催得急，先跳过" | 错误代码比晚交付代价更高。修复成本是预防的 10x |
| "这个检查太严格了" | 检查标准是用户确认的。降低标准需要用户授权 |
| "我可以在后面补上" | 「后面」永远不会来。现在就做 |
| "之前的代码也是这样的" | 之前的代码可能也有问题。不要复制错误 |

---

## 场景专用表格

### Phase 1 Plan Check 门禁

| 你可能在想 | 实际情况 |
|-----------|----------|
| "任务表看起来没问题" | 「看起来」不够。必须用 tech-plan-checker 验证 |
| "依赖关系很简单" | 简单依赖也可能有环。拓扑排序不骗人 |
| "先开始写代码，边写边改任务表" | 计划先行，执行在后。没有通过的计划 = 没有方向 |

### Phase 2 Wave Execution 门禁

| 你可能在想 | 实际情况 |
|-----------|----------|
| "这个 Wave 只有一个任务，不需要门禁" | 一个任务的 Wave 也可能编译失败 |
| "覆盖率差一点没关系" | 差一点的覆盖率 = 未被测试的边界条件 = 生产 Bug |
| "先继续下一个 Wave，回头再补测试" | 不会回头。技术债只增不减 |

### Phase 3 Code Review 门禁

| 你可能在想 | 实际情况 |
|-----------|----------|
| "我自己写的代码，我很清楚" | 自己审查自己的代码 = 护短。必须用独立 Subagent |
| "安全审查太浪费时间" | 一个 SQL 注入漏洞 = 一次安全事故 = 职业生涯污点 |
| "这个改动很小，不需要审查" | 最危险的 Bug 往往藏在最小的改动里 |
| "三步审查太慢了，能不能并行" | 方向错了，安全和质量审查都是浪费。顺序不可变 |

### Phase 4 Verification 门禁

| 你可能在想 | 实际情况 |
|-----------|----------|
| "代码编译通过就说明没问题" | 编译通过 ≠ 功能正确。占位代码也能编译 |
| "测试全绿就说明没问题" | 测试全绿可能是测试写得太宽松。需要覆盖率数据 |
| "技术方案可能过时了" | 过时 = 需要更新方案 + 用户确认，不是忽略方案 |

## 使用方式

在各 SKILL.md 的 HARD-GATE 后面引用本文件：

```markdown
<HARD-GATE>
... 门禁内容 ...
</HARD-GATE>

<ANTI-RATIONALIZATION>
在跳过上述检查之前，先问自己：是不是在找借口？
详见 `anti-rationalization.md` 中的合理化表格。
</ANTI-RATIONALIZATION>
```
