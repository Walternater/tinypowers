# anti-rationalization.md

## 作用

这份文档用来对抗一个很常见的执行风险：在赶进度、上下文变长或刚修完问题时，Agent 会倾向于替自己找理由，绕过本来应该执行的检查。

`anti-rationalization` 的目标不是增加新门禁，而是提醒执行中的 Agent：
- 不要把“感觉没问题”当成“已经验证”
- 不要把“先做了再说”当成“可控风险”
- 不要把“我自己刚写的”当成“我能客观看待”

## 使用方式

在关键门禁附近引用这份文档，作为自检提示：

```markdown
<ANTI-RATIONALIZATION>
在跳过上述检查之前，先问自己：是不是在找借口？
详见 `anti-rationalization.md`。
</ANTI-RATIONALIZATION>
```

它通常出现在：
- Plan Check 之前
- Wave 末尾质量门禁之前
- 顺序审查之前
- 最终 Verification 之前

## 通用提醒

| 你可能在想 | 更可靠的判断 |
|-----------|--------------|
| 这只是个小改动 | 小改动同样可能破坏边界条件 |
| 我已经检查过了 | 自查不等于独立验证 |
| 用户催得急，先跳过 | 带着已知风险继续推进，返工成本通常更高 |
| 这一步应该不会出问题 | “应该”不是证据，跑完检查才是 |
| 我后面再补 | 没进入当前流程的补做，通常不会自动发生 |

## Phase 1: Plan Check

| 你可能在想 | 更可靠的判断 |
|-----------|--------------|
| 任务表看起来没问题 | 任务表必须经过 `tech-plan-checker` 验证 |
| 依赖关系很简单，不会有错 | 简单依赖也可能有遗漏或环 |
| 先开始写，边写边改任务表 | 没过 Plan Check 的执行，很容易把错误放大 |

## Phase 2: Wave Execution

| 你可能在想 | 更可靠的判断 |
|-----------|--------------|
| 这个 Wave 很小，不需要门禁 | 只要进入下一 Wave，就应该先确认当前 Wave 可收敛 |
| 覆盖率差一点没关系 | 差一点通常意味着边界场景还没被证明 |
| 先继续，测试回头补 | 下一 Wave 会继续堆上下文，补测意愿只会更低 |

## Phase 3: 顺序审查

| 你可能在想 | 更可靠的判断 |
|-----------|--------------|
| 这是我自己写的，我最清楚 | 正因为是自己写的，更容易护短 |
| 安全审查可以最后一起看 | 高危安全问题不应被带入后续阶段 |
| 三步审查太慢了，能不能并行 | 如果方案本身没实现对，后两步的结论没有意义 |

## Phase 4: Verification

| 你可能在想 | 更可靠的判断 |
|-----------|--------------|
| 能编译就算完成 | 编译通过只证明能构建，不证明功能闭环 |
| 测试全绿就够了 | 测试质量和覆盖范围也要看 |
| 方案可能已经过时，可以先忽略 | 方案变更需要回到文档和决策层处理 |

## 判断标准

当你想跳过某一步时，可以先问自己两个问题：
- 我现在手里的是证据，还是只是判断？
- 如果恢复到新会话，这个跳过动作还能被别人理解和接受吗？

只要这两个问题里有一个答不上来，就不该跳过。
