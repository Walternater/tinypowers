# context-preload.md

## 作用

这份文档定义 `/tech:code` 在 Wave Execution 阶段如何通过上下文预加载减少重复文件读取，并如何生成自包含的 Per-Task 命令文件以实现 token 优化。

## 核心原则

- 编排层读一次，传递给所有 subagent
- subagent 收到预加载的上下文后，不应重新读取
- 预加载的内容只包含当前任务需要的最小集合
- 每个任务生成独立的命令文件，内嵌完整上下文，消除重复读取

## 两种上下文注入模式

### 模式 A：Per-Task 命令文件（推荐）

为每个任务生成独立的命令文件，内嵌所有必要上下文：

```
features/{id}/
├── commands/                          # Per-task 命令文件
│   ├── T-001-execute.md               # T-001 的完整执行上下文
│   ├── T-002-execute.md
│   └── ...
```

**优点：**
- subagent 只需读取一个文件即可获得所有上下文
- 避免重复读取同一文档的多个 subagent 之间互相覆盖
- 上下文精确匹配任务需要，无冗余

### 模式 B：Prompt 内嵌（备选）

每个 subagent 的任务 prompt 中，在具体任务描述之前，插入预加载区块：

```
--- 预加载上下文 START ---

## 技术方案摘要
{从 技术方案.md 提取的核心内容}

## 当前任务上下文
- Feature: {id}
- 当前 Wave: {wave_number}
- 已锁定决策: {从 技术方案.md 决策表提取}

## 相关接口 / 数据结构
{从 技术方案.md 提取与当前任务相关的接口定义}

--- 预加载上下文 END ---

重要：以上上下文已预加载，你不需要重新读取这些文件。
只读取你需要修改的目标文件。
```

## 预加载协议（模式 A 为主）

### Step 1: 编排层收集上下文

在启动任何 Wave 之前，编排层一次性读取以下文件：

```text
必读（所有 Wave 共享，只读一次）:
  features/{id}/技术方案.md
  features/{id}/任务拆解表.md
  features/{id}/SPEC-STATE.md
  features/{id}/STATE.md
  features/{id}/notepads/learnings.md     (如果存在)

按需读（每个 Wave 开始时读一次）:
  features/{id}/需求理解确认.md
```

### Step 2: 生成 Per-Task 命令文件

为 Wave 中的每个任务生成 `{T-XXX}-execute.md`：

```bash
mkdir -p features/{id}/commands/
```

每个文件内嵌（见下方 Per-Task 文件格式）。

### Step 3: subagent 的读取约束

subagent 收到 Per-Task 文件后：
- 禁止重新读取已预加载的文件（技术方案、任务表等）
- 只读取需要修改的目标源码文件
- 如果预加载内容与实际文件有冲突，以实际文件为准并报告偏差

## Per-Task 命令文件格式

```markdown
# T-001: 实现用户注册接口

<!-- AUTO-GENERATED by /tech:code. DO NOT reload context above. -->

## 任务目标

实现用户注册 REST API，包含邮箱验证和密码加密存储。

## 验收标准

- [ ] POST /api/users 返回 201，body 含 userId
- [ ] 邮箱格式无效返回 400
- [ ] 密码使用 BCrypt 加密（强度 >= 12）
- [ ] 重复邮箱返回 409
- [ ] UserServiceTest 行覆盖率 >= 80%

## 涉及文件

- `src/main/java/com/example/service/UserService.java`
- `src/main/java/com/example/controller/UserController.java`
- `src/main/java/com/example/model/User.java`

## 技术方案片段

### 接口契约
- POST /api/users
- Request: `{ "email": "...", "password": "...", "username": "..." }`
- Response 201: `{ "userId": "uuid", "email": "...", "createdAt": "..." }`

### DDL
```sql
CREATE TABLE users (
  id VARCHAR(36) PRIMARY KEY,
  email VARCHAR(255) UNIQUE NOT NULL,
  password_hash VARCHAR(255) NOT NULL,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### 锁定决策
- D-01: 使用 BCrypt 而不是 SCrypt
- D-03: 用户名允许字母数字和下划线，3-30字符

## 已积累的智慧 (from learnings.md)

> BCrypt 初始化问题：确保 JCE Unlimited Strength Jurisdiction Policy Files 已安装。

<!-- DO NOT reload context above. All needed context is embedded. -->
```

## 裁剪规则

预加载不是全量塞入。按以下规则裁剪：

| 内容类型 | 裁剪规则 |
|---------|---------|
| 技术方案全文 | 只注入与当前任务相关的章节 |
| 接口定义 | 只注入当前任务涉及的接口 |
| 数据库设计 | 只注入当前任务涉及的表 |
| 决策记录 | 全量注入（决策是全局约束） |
| learnings | 只注入与当前任务类型相关的条目 |

## Context 估算

在生成 per-task 文件时，编排层应估算每个任务 prompt 的总 context 量：

| 任务类型 | 典型 context 占用 |
|---------|-----------------|
| 单文件修改 | ~5-10% |
| 多文件新增 | ~10-20% |
| DB 变更 | ~15-25% |
| 完整模块 | ~20-35% |

如果单个 Wave 内所有任务总 context 占用预计超过 50%，应触发拆分提示。

## 适用范围

- **适用**：Wave Execution 中的 subagent 任务分发（优先使用 Per-Task 命令文件）
- **不适用**：审查阶段的 agent（审查需要独立读取完整文件）
- **不适用**：tech-feature 的各 Phase

## 与审查阶段的关系

审查阶段（spec-compliance-reviewer、security-reviewer、code-reviewer）必须独立读取文件，不能依赖预加载上下文。原因是：
- 审查的独立性要求：审查者不能信任被审查者提供的上下文
- 审查需要看完整文件，不是摘要
- 预加载可能遗漏审查者需要的细节
