# deviation-handling.md

## 偏差处理策略

本文档描述执行过程中发现偏差时的处理机制。

---

## 偏差分类

| 类型 | 定义 | 影响 | 处理策略 |
|------|------|------|----------|
| **Scope 偏差** | 任务范围变化 | 低 | 记录，继续执行 |
| **依赖偏差** | 实际依赖与预期不符 | 中 | 自动补充依赖，重新分波 |
| **技术偏差** | 技术方案无法实现 | 高 | 暂停，询问用户 |
| **资源偏差** | 资源不可用（API/DB等） | 中 | 重试或跳过 |

---

## 偏差处理流程

```
┌─────────────────────────────────────┐
│  Step 1: 偏差检测                   │
│  执行中发现异常/错误                 │
└──────────────────┬────────────────┘
                   ↓
┌─────────────────────────────────────┐
│  Step 2: 偏差分类                   │
│  判断偏差类型                       │
└──────────────────┬────────────────┘
                   ↓
┌─────────────────────────────────────┐
│  Step 3: 处理策略选择               │
│  根据类型选择处理方式               │
└──────────────────┬────────────────┘
                   ↓
┌─────────────────────────────────────┐
│  Step 4: 执行处理                  │
│  记录 → 修复/暂停/继续              │
└──────────────────┬────────────────┘
                   ↓
┌─────────────────────────────────────┐
│  Step 5: 更新任务表                 │
│  同步偏差信息到任务表               │
└─────────────────────────────────────┘
```

## Scope 偏差

### 定义

任务执行中发现需求范围变化，如：
- 用户要求增加一个字段
- 需求文档中未提及的功能点

### 处理策略

```
IF Scope 偏差 THEN
  1. 记录偏差详情到日志
  2. 评估影响范围
  3. 询问用户是否纳入本迭代

  IF 用户同意纳入 THEN
    补充任务到任务表
    重新分析依赖
  ELSE
    记录为"后续迭代项"
  END
END
```

### 记录格式

```markdown
## Scope 偏差记录

**时间**: 2026-03-27 15:00
**Task**: T-004
**发现者**: Agent
**描述**: 用户要求增加"记住手机号"功能

**影响评估**:
  - 新增字段: remember_phone (Boolean)
  - 涉及: T-004, T-005

**处理结果**:
  ☐ 纳入本迭代（需补充任务）
  ☑ 延期至V2

**用户确认**: ____
```

## 依赖偏差

### 定义

任务执行中发现实际依赖与预期不符，如：
- 预期无依赖，实际需要某接口完成
- 预期依赖A，实际还需B完成

### 处理策略

```
IF 依赖偏差 THEN
  1. 识别实际依赖关系
  2. 更新任务表中的依赖
  3. 重新分波
  4. 如果当前Wave任务受影响，调整执行计划

  自动处理，无需用户介入
END
```

### 自动修复示例

```
检测到依赖偏差：

原任务表：
  T-004 (LoginController) 依赖 T-003 (LoginService)

实际情况：
  T-004 还需要 T-007 (UserService) 完成

自动修复：
  更新依赖：T-004 → [T-003, T-007]
  重新分波：
    Wave 4: [T-007] (新增)
    Wave 5: [T-004] (调整后)
  新增 Task: T-007 (获取用户信息)

报告：依赖偏差已自动修复，详见调整记录
```

## 技术偏差

### 定义

技术方案无法实现，如：
- 选型技术有已知bug
- 第三方API不可用
- 性能目标无法达成

### 处理策略

```
IF 技术偏差 THEN
  1. 暂停当前任务
  2. 记录偏差详情
  3. 提出替代方案
  4. 等待用户决策

  示例输出：

  ---
  ⚠️  技术偏差检测

  **Task**: T-010
  **问题**: 选择的短信API有频率限制，无法满足需求

  **替代方案**:

  方案A: 换用另一短信API (阿里云)
    优点: 限制宽松，性能好
    缺点: 需要重新对接

  方案B: 降低发送频率
    优点: 不用换API
    缺点: 影响用户体验

  请选择方案，或提供其他方案：____
  ---
END
```

## 资源偏差

### 定义

资源不可用，如：
- 第三方API超时/不可用
- 数据库连接失败
- 依赖的微服务不可用

### 处理策略

```
IF 资源偏差 THEN
  IF 可重试 THEN
    # 等待后重试
    WAIT 30s
    RETRY (最多3次)
  END

  IF 不可重试 OR 重试失败 THEN
    # 降级处理
    执行降级逻辑
    记录偏差
    继续执行
  END
END
```

```json
  "user_confirmed": true|false,
  "resolution_detail": "详细处理说明"
}
```
