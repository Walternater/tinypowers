# deviation-handling.md

## 作用

偏差处理定义了 `/tech:code` 在执行过程中发现“现实和计划不一致”时该怎么做。

它的目标不是把所有异常都自动吞掉，而是区分：
- 哪些可以自动收敛
- 哪些必须停下来
- 哪些已经升级到架构层问题

## 偏差类型

| 类型 | 含义 | 默认动作 |
|------|------|----------|
| Scope 偏差 | 需求范围发生变化 | 记录并请用户确认是否纳入当前迭代 |
| 依赖偏差 | 实际依赖与任务表不一致 | 更新依赖并重新分 Wave |
| 技术偏差 | 技术方案在当前条件下无法落地 | 暂停并提出替代方案 |
| 资源偏差 | 外部资源暂时不可用 | 重试、降级或记录后继续 |

## 处理顺序

发生偏差时，按以下顺序处理：

```text
识别偏差
  -> 分类
  -> 评估影响
  -> 记录到 STATE.md
  -> 选择自动修复 / 暂停确认 / 升级处理
```

## 1. Scope 偏差

典型场景：
- 用户新增字段或功能点
- 原需求没有写，但执行时发现必须额外补充范围

处理方式：
- 先记录影响范围
- 明确是纳入本次 Feature 还是留到后续迭代
- 如果纳入当前迭代，补充任务并重新分析依赖

不要在没有确认的情况下，私自把新需求混进当前交付。

## 2. 依赖偏差

典型场景：
- 原以为任务 A 只依赖 B，实际还依赖 C
- 原任务表缺了一个前置任务

处理方式：
- 修正依赖关系
- 必要时补充新任务
- 重新分 Wave
- 把变更原因写进 `STATE.md`

这类偏差通常可以自动处理，但不能悄悄处理。

## 3. 技术偏差

典型场景：
- 选型组件不满足目标性能
- 第三方 API 有硬限制
- 技术方案的关键假设不成立

处理方式：
- 暂停当前方向
- 写清问题和影响
- 至少给出两个替代方案
- 等待用户或团队决策

技术偏差不是普通 bug，不能靠不断微调实现去硬扛。

## 4. 资源偏差

典型场景：
- 外部接口超时
- 数据库或依赖服务暂时不可用
- 本地测试环境缺资源

处理方式：
- 可重试的先有限重试
- 可降级的走降级方案
- 无法收敛的则记录阻塞并暂停相关任务

## 记录方式

偏差至少应记录这些信息：
- 时间
- 所在 Wave
- 相关任务 ID
- 偏差类型
- 影响描述
- 采取的处理动作
- 是否需要用户确认

可以直接记录在 `STATE.md` 的“偏差”章节，必要时再补充独立文件到 `features/{id}/deviations/`。

## 3 次失败 -> 质疑架构

这是偏差处理里最重要的升级规则。

如果同一问题连续失败 3 次，就不要再继续第 4 次同方向修补。此时应默认怀疑：
- 技术方案的假设错了
- 模块边界拆错了
- 数据模型设计错了
- 当前任务拆解方式不适合实现

### 什么算同一问题

- 同一个测试连续失败 3 次
- 同一个编译错误反复出现
- 同一个审查意见重复被打回
- 同一种占位或空实现问题反复出现

### 升级后的动作

- 停止当前修补路径
- 回顾设计假设
- 给出架构层调整方向
- 把失败摘要和判断写进 `STATE.md`
- 等待进一步决策

## 偏差日志格式

偏差记录直接写入 STATE.md 的"偏差"章节，格式：

```markdown
| 时间 | 任务ID | 规则 | 描述 | 处理结果 |
|------|--------|------|------|----------|
| HH:MM | T-XXX | Rule N | 简述 | 已修复/已升级/待确认 |
```

升级到 Rule 4 时，在 STATE.md 记录升级原因和待确认事项。

## 判断标准

一个好的偏差处理结果应该做到：
- 问题被看见，而不是被掩盖
- 自动修复只发生在低风险偏差上
- 需要决策的问题被及时上升
- 后续会话能从 `STATE.md` 读懂发生过什么

## Gotchas

- **同一 Wave 连续 3 次失败后继续同方向尝试**：换方向不算重置：同一问题连续 3 次失败后必须升级，转入架构质疑
