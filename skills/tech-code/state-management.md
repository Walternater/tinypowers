# state-management.md

## 作用

`STATE.md` 是 `/tech:code` 执行期间的唯一真相源，用来回答“现在做到哪了、下一步是什么、能不能恢复”。

它不是归档文件，也不是复盘报告，而是执行期的活跃状态。

## 文件位置

```text
features/{id}/STATE.md
```

每个 Feature 独立维护自己的状态文件。

## 核心原则

- 文件即状态：状态要写进文件，而不是只留在上下文里
- 实时更新：关键节点立即更新，不靠事后补记
- 只写摘要：记录足够恢复的信息，不堆执行细节
- 单一真相源：恢复和推进都以 `STATE.md` 为准

## 建议结构

```markdown
# STATE: {Feature ID}

> 最后更新: {datetime} | 当前阶段: {phase}

## 位置
- 当前技能: tech-code
- 当前阶段: Phase 2 - Wave Execution
- 当前 Wave: 2 / 5

## 决策
| ID | 内容 | 确认人 | 日期 |
|----|------|--------|------|
| D-01 | ... | ... | ... |

## 进度
### Wave 1 DONE
- [x] T-001 ... (智慧已积累)

### Wave 2 IN_PROGRESS
- [ ] T-002 ...

### Wave 3 PENDING
- [ ] T-003 ...

## 阻塞
- 无

## 偏差
- 无

## 上次操作
- Wave 1 门禁通过，智慧已积累到 learnings.md，开始 Wave 2
```

## 必须更新的时机

以下节点必须更新 `STATE.md`：
- `/tech:code` 启动时
- 每个任务完成或失败时
- 每个 Wave 完成时
- 阶段切换时
- 出现阻塞时
- 出现偏差时
- 用户确认新决策时

如果没有写进 `STATE.md`，就等于没有真正纳入流程。

## 必须读取的时机

以下节点必须先读取 `STATE.md`：
- `/tech:code` 启动时
- Session 恢复时
- 启动下一 Wave 前
- 进入审查前
- 进入 Verification 前

## 与 Snapshot 的关系

`STATE.md` 和 Snapshot 不是两套并行状态系统，它们分工不同：

| 组件 | 位置 | 职责 |
|------|------|------|
| `STATE.md` | `features/{id}/` | 执行期主状态，长期保留 |
| Snapshot | `/tmp/` | 跨会话提示和恢复入口，临时存在 |

恢复时应该这样理解：
- Snapshot 负责提醒“有未完成工作”
- `STATE.md` 负责说明“具体做到哪里了”

## 压缩规则

`STATE.md` 应尽量控制在约 100 行以内。

当已完成 Wave 太多时，可以把旧 Wave 压缩成摘要，例如：

```markdown
### Wave 1 DONE
T-001, T-002, T-003 (Gate OK)
```

但以下内容不要压掉：
- 当前 Wave
- 未完成任务
- 阻塞项
- 偏差项
- 最近一次操作

## 禁止事项

- 禁止未完成就把任务标成 `[x]`
- 禁止跳过更新直接推进下一 Wave
- 禁止删除已经确认的决策记录
- 禁止让 Snapshot 和 `STATE.md` 各写一套不同结论

## 判断标准

一份合格的 `STATE.md`，应该让新的会话在不依赖旧上下文的情况下，也能知道：
- 当前在哪个阶段
- 当前 Wave 是多少
- 哪些任务已经完成
- 哪些问题还没解决
- 下一步应该做什么
