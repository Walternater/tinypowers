# documenter-guide.md

## 作用

这份文档定义 `/tech:commit` 在提交前如何同步代码相关文档，避免“实现已经变了，说明还停在旧版本”。

## 核心原则

- 文档服务于交付，不是装饰物
- 只更新真正受影响的文档
- 文档描述当前实现，不描述理想中的实现
- 文档同步应发生在验证完成之后、正式提交之前

## 常见需要同步的文档

| 文档 | 触发条件 |
|------|----------|
| `features/{id}/技术方案.md` | 实现细节、验收状态或决策落地发生变化 |
| API 文档 | Controller、路由、请求响应结构变更 |
| README | 用法、入口、配置、能力边界发生变化 |
| 数据库文档 | 表结构、字段、索引、迁移方式变更 |
| 部署文档 | 启动方式、环境变量、依赖服务变更 |

## 同步顺序

建议按这个顺序检查：

```text
实现变更
  -> 技术方案
  -> 接口/数据结构文档
  -> README / 部署说明
```

先改离实现最近的文档，再决定是否需要更新面向外部的入口文档。

## 1. 技术方案同步

重点不是重写方案，而是补齐“这次实际怎么落地了”。

建议至少检查：
- 功能点状态是否需要标记为已完成
- 决策是否已经被实现验证
- 验收标准是否已经落地
- 是否需要补充实现记录或风险说明

## 2. API 文档同步

当接口发生变化时，需要同步：
- 路径和方法
- 请求参数
- 响应结构
- 错误码或异常行为
- 示例请求和响应

如果项目把 API 文档写在 `技术方案.md` 中，就在那里同步；如果有独立 OpenAPI 或接口文档，也要保持一致。

## 3. README 同步

README 只在对使用者有感知变化时更新，例如：
- 新增能力
- 新增配置
- 接入方式变化
- 本地启动步骤变化

不要把纯内部实现细节塞进 README。

## 4. 数据库或部署文档同步

当本次改动影响运行环境时，至少要说明：
- 新增或修改的表结构
- 数据迁移要求
- 新增环境变量
- 新的外部依赖
- 启动或部署步骤变化

## 验证清单

文档同步完成后，至少检查：
- 文档中的接口签名和代码一致
- 文档中的状态描述和实际交付一致
- 没有继续引用旧路径或旧命令
- 没有把未完成项写成已完成
- 没有明显拼写或结构错误

## 提交建议

如果代码和文档强绑定，可以一起提交。

如果文档变更较多、需要单独阅读，也可以拆成单独的 `docs(...)` commit。关键是让历史可读，而不是强行追求固定拆法。
