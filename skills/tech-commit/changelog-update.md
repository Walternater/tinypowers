# changelog-update.md

## 作用

这份文档定义什么时候应该更新 `CHANGELOG.md`，以及更新时应该记录什么。

## 核心原则

- changelog 记录的是“对项目有意义的变化”，不是全部提交历史
- 优先面向使用者、维护者和发布说明阅读者
- 同一类变化尽量聚合，不把零碎过程噪音直接搬进去

## 什么时候更新

通常在以下场景更新：
- 仓库明确维护 `CHANGELOG.md`
- 本次变更会出现在后续版本说明中
- 新增、修复或变更会被项目使用者感知

以下场景通常不必强制更新：
- 纯内部重构
- 过程性提交
- 不对外暴露的微小整理

## 建议结构

如果仓库采用 Keep a Changelog 风格，通常使用：

```markdown
## [Unreleased]

### Added
- ...

### Changed
- ...

### Fixed
- ...

### Security
- ...
```

## 从改动到分类的映射

| 改动类型 | 常见归类 |
|----------|----------|
| 新功能 | `Added` |
| 行为调整 | `Changed` |
| 缺陷修复 | `Fixed` |
| 安全修复 | `Security` |
| 弃用或移除 | `Deprecated` / `Removed` |

不要机械地按 commit type 逐条照抄，应该按“读者关心的变化”来整理。

## 更新方法

更新 changelog 时，建议先做三件事：
- 回顾本次交付真正产生了哪些用户可感知变化
- 合并重复或细碎描述
- 去掉只对实现过程有意义的表述

写法上尽量做到：
- 一条只写一件事
- 先写结果，再写范围
- 避免只写 issue 编号而不写内容

## 版本发布时

如果项目有正式发布流程，通常会把 `Unreleased` 中的内容整理到具体版本号下，并补上发布日期。

这一步的重点是：
- 确认条目没有重复
- 确认分类合理
- 确认发布版本真正包含这些变化

## 判断标准

一份好的 changelog 更新应当让读者快速知道：
- 这个版本新增了什么
- 修掉了什么问题
- 有没有破坏性变化
- 是否有值得特别关注的安全或兼容性信息
