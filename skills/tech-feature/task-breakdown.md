# task-breakdown.md

## 作用

本文档说明 `/tech:feature` 如何把技术方案拆成可执行任务。

目标不是生成一张“看起来很完整”的表，而是为 `/tech:code` 提供真正可执行的输入。

## 推荐拆解层级

```text
Epic -> Story -> Task
```

理解方式：
- Epic 表示较大的功能块
- Story 表示可独立理解和验收的子目标
- Task 表示可执行的具体工程动作

## 拆解原则

### Epic

- 同一 Epic 内功能应高相关
- 应具备相对独立的验收意义

### Story

- 应可独立开发
- 应可独立测试
- 应有明确验收标准

### Task

- 粒度应适中
- 应有清晰类型
- 应有依赖关系
- 应可验证是否完成

## 验收标准

每个 Story 或关键 Task 都应有可验证标准。

不要只写：
- “功能正常”
- “联调通过”

更好的写法是：
- 指定具体行为
- 指定错误场景
- 指定关键边界

## 任务表建议字段

| 字段 | 说明 |
|------|------|
| 任务编号 | 如 `T-001` |
| 任务名称 | 简洁明确 |
| 类型 | 后端 / 前端 / DB / 测试 |
| 估时 | 用于后续规划 |
| 依赖 | 明确前置任务 |
| 验收标准 | 可验证 |

## 依赖管理

至少明确：
- 技术依赖
- 业务依赖
- 必要的人力或协作依赖

原则：
- 尽量并行
- 明确关键路径
- 不要把隐式依赖留给执行阶段再猜

## 估时建议

可选方式：
- 类比估算
- 三点估算
- T-shirt 尺寸估算

如果一个任务已经大到难以估算，通常说明还该继续拆。

## 一个好的任务拆解是什么样

读完任务拆解表后，团队应该能回答：

1. 先做什么、后做什么？
2. 哪些任务可以并行？
3. 哪些任务做完算真的完成？
4. 后续 `/tech:code` 是否可以直接接手执行？

## Gotchas

- **任务粒度过大**：把"实现订单模块"当一个 Task → 执行时无法评估进度：Task 必须 ≤ 1 人天（8h）
