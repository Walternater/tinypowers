# ambiguity-check.md

## 歧义检测环节

在需求分析和方案设计之间，必须执行歧义检测，确保需求清晰无歧义。

---

## 执行时机

```
需求理解（Phase 1）完成之后
技术方案设计（Phase 3）开始之前
```

---

## 检测维度

### 1. 模糊描述词

常见模糊词需要明确定义：

| 模糊词 | 需要澄清 | 可能的定义 |
|--------|----------|------------|
| "快速" | 响应时间？ | < 1s / < 200ms / < 100ms |
| "友好" | 什么样的体验？ | 操作步骤少 / 错误提示清晰 |
| "高效" | 效率提升多少？ | 减少 X 步操作 / 耗时减少 Y% |
| "安全" | 什么级别的安全？ | 密码加密 / 数据脱敏 / 权限校验 |
| "可靠" | SLA 要求？ | 99.9% / 99.99% |
| "实时" | 延迟要求？ | < 1s / < 5s / < 30s |

### 2. 边界条件

检查以下边界是否明确：

| 边界类型 | 检查项 |
|----------|--------|
| 数量边界 | 最大用户数？最大数据量？最大并发？ |
| 时间边界 | 超时时间？缓存过期时间？会话超时？ |
| 长度边界 | 字符串最大长度？列表最大长度？ |
| 频率边界 | 请求频率限制？操作频率限制？ |

### 3. 异常场景

检查以下异常是否覆盖：

| 异常类型 | 示例 |
|----------|------|
| 输入异常 | 空输入、非法格式、超长输入 |
| 网络异常 | 超时、断网、重试 |
| 业务异常 | 余额不足、权限不足、状态不对 |
| 系统异常 | 服务宕机、数据库不可用、缓存失效 |

### 4. 数据量级

检查以下量级是否明确：

| 量级类型 | 检查项 |
|----------|--------|
| 用户量级 | 日活、月活、注册用户总数 |
| 数据量级 | 单表数据量、日增量 |
| 并发量级 | 峰值QPS、同时在线人数 |

```markdown
# 待澄清问题列表

## 高优先级（阻塞）

|------|--------|----------|------|----------|

## 歧义处理流程

```
┌─────────────────────────────────────┐
│  Step 1: 识别歧义                   │
│  扫描PRD，标记模糊点                 │
└──────────────────┬──────────────────┘
                   ↓
┌─────────────────────────────────────┐
│  Step 2: 评估影响                   │
│  高/中/低优先级分类                 │
└──────────────────┬──────────────────┘
                   ↓
┌─────────────────────────────────────┐
│  Step 3: 生成澄清问题                 │
│  提出具体问题和建议定义               │
└──────────────────┬──────────────────┘
                   ↓
┌─────────────────────────────────────┐
│  Step 4: 用户确认                    │
│  逐项确认，接受建议或提供新定义       │
└──────────────────┬──────────────────┘
                   ↓
┌─────────────────────────────────────┐
│  Step 5: 更新需求理解                 │
│  将澄清结果并入需求文档               │
└─────────────────────────────────────┘
```

## 澄清对话示例

### 对话1：性能歧义

```
歧义：PRD 中提到"登录响应要快"

我：
---
❓ 澄清问题 #1

PRD 描述："登录响应要快"

问题："快"的定义不明确，请确认以下选项或补充：
  A. < 1 秒
  B. < 500 毫秒
  C. < 200 毫秒

如果以上都不合适，请提供具体要求：____

---

用户回复：B (< 200 毫秒)

✅ 已记录：登录响应 < 200ms
```

### 对话2：频率歧义

```
歧义：PRD 中提到"防止频繁登录"

我：
---
❓ 澄清问题 #2

PRD 描述："防止频繁登录尝试"

问题："频繁"的定义不明确，请确认：
  A. 1 分钟内超过 5 次
  B. 5 分钟内超过 10 次
  C. 10 分钟内超过 20 次

建议：B（5分钟内超过10次）

请确认或提供自定义定义：____

---

用户回复：A

✅ 已记录：1分钟内超过5次登录尝试，触发验证码
```

## 检测结论

✅ **可以进入技术方案设计**

理由：
- 无高优先级歧义未澄清
- 已识别的歧义均有建议定义
- 用户可接受当前理解或提供新定义
```

---

## 关键原则

1. **高优先级歧义必须澄清**才能进入方案设计
2. **每个歧义提供建议定义**减少用户决策成本
3. **记录而非强制**用户可拒绝建议提供自己的定义
4. **更新需求文档**澄清后的定义需要同步到 PRD
