# requirements-guide.md

## 作用

本文档说明 `/tech:feature` 在"需求理解"阶段应该如何读 PRD、怎么问问题、怎么形成确认稿。

## 路由差异化检查

不同复杂度路由需要确认的内容不同：

| 维度 | Fast | Medium | Standard |
|------|------|--------|----------|
| 需求背景 | ✅ | ✅ | ✅ |
| 功能范围 | ✅ | ✅ | ✅ |
| 验收标准 | ✅ | ✅ | ✅ |
| 目标用户 | ❌ | ✅ | ✅ |
| 非功能需求 | ❌ | 可选 | ✅ |

**Fast 路由只需确认 3 项：背景、范围、验收标准。**

## 推荐检查顺序

### 1. 需求背景

先问清：
- 为什么做这个功能
- 解决什么业务问题
- 预期价值是什么

### 2. 目标用户（Medium / Standard）

再确认：
- 谁会使用
- 在什么场景下使用
- 使用频率和重要性如何

### 3. 功能范围

至少区分：
- 必做
- 可选
- 明确不做

### 4. 验收标准

重点确认：
- 是否包含具体的 HTTP 状态码或响应字段
- 是否覆盖正向、反向、边界、异常场景
- 是否可自动化测试（不允许"功能正常"等模糊描述）

> EARS 格式是推荐写法，但不强求。

### 5. 非功能需求（Standard）

重点看：
- 性能
- 安全
- 兼容性
- 可观测性

## 推荐输出

需求理解确认至少应包含：

```text
需求名称
需求链接

一、我的理解
- 背景
- 目标
- 核心功能点

二、我的疑问
- 待澄清问题

三、补充建议
- 可选的技术建议或范围建议
```

## 交互原则

批量提问，按需追问：
- 一轮提出所有核心维度的问题
- 用户自由回答，不限格式
- AI 只追问用户回答中缺失或矛盾的关键项
- 不需要逐项确认，不需要"one question at a time"

## 需求理解阶段的判断标准

如果下面这些问题都能回答清楚，通常就可以进入歧义检测（Standard）或技术方案（Fast/Medium）：

1. 为什么做这个需求？
2. 本次必须交付什么？
3. 完成后如何验收？
