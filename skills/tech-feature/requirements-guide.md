# requirements-guide.md

## 作用

本文档说明 `/tech:feature` 在“需求理解”阶段应该如何读 PRD、怎么问问题、怎么形成确认稿。

## 需求理解要回答什么

至少确认这五类内容：
- 需求背景
- 目标用户
- 功能范围
- 验收标准
- 非功能需求

## 推荐检查顺序

### 1. 需求背景

先问清：
- 为什么做这个功能
- 解决什么业务问题
- 预期价值是什么

### 2. 目标用户

再确认：
- 谁会使用
- 在什么场景下使用
- 使用频率和重要性如何

### 3. 功能范围

至少区分：
- 必做
- 可选
- 明确不做

### 4. 验收标准

重点确认：
- 是否包含具体的 HTTP 状态码或响应字段
- 是否覆盖正向、反向、边界、异常场景
- 是否可自动化测试（不允许"功能正常"等模糊描述）

> EARS 格式是推荐写法，但不强求。

### 5. 非功能需求

重点看：
- 性能
- 安全
- 兼容性
- 可观测性

## 推荐输出

需求理解确认至少应包含：

```text
需求名称
需求链接

一、我的理解
- 背景
- 目标
- 核心功能点

二、我的疑问
- 待澄清问题

三、补充建议
- 可选的技术建议或范围建议
```

## 交互原则

采用 one question at a time：
- 每次只确认一个主题
- 优先确认会影响方案设计的核心问题
- 不要把多个大问题揉在一个回合里

## 需求理解阶段的判断标准

如果下面这些问题都能回答清楚，通常就可以进入歧义检测：

1. 为什么做这个需求？
2. 谁是主要用户？
3. 本次必须交付什么？
4. 完成后如何验收？
5. 是否存在明显非功能约束？

## Gotchas

- **不探索上下文直接设计**：对现有代码结构不熟悉就提出方案 → 可能与现有模式冲突或重复造轮子：Phase 1 必须先了解项目上下文
