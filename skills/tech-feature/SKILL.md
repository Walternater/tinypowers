# /tech:feature

## 功能
需求分析 → 歧义检测 → 技术方案 → 任务拆解

**核心特性：**
- 歧义检测环节（One question at a time）
- Scope → Epics → Stories 分层任务分解
- 每个阶段验证清单
- 关键节点人工确认

## 执行流程

```
┌─────────────────────────────────────────────────────┐
│  Phase 0: 准备                                      │
│  - 解析需求ID/标题                                   │
│  - 确认增量还是全新功能                               │
│  - 创建feature目录和代码分支                          │
└────────────────────────┬────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────┐
│  Phase 1: 需求理解                                   │
│  - 用户创建PRD文档                                   │
│  - AI分析需求背景                                   │
│  - 逐项确认理解（One question at a time）            │
└────────────────────────┬────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────┐
│  Phase 2: 歧义检测                                   │
│  - 识别模糊点                                       │
│  - 生成澄清问题                                      │
│  - 输出需求理解确认文档                              │
└────────────────────────┬────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────┐
│  Phase 3: 技术方案                                   │
│  - 架构设计                                         │
│  - 接口设计                                         │
│  - 数据库设计                                        │
│  - 方案评审检查                                     │
└────────────────────────┬────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────┐
│  Phase 4: 任务拆解 ★ 关键确认点                     │
│  - Scope → Epics → Stories 分解                     │
│  - 每个Story含验收标准                              │
│  - 输出任务拆解表                                    │
└────────────────────────┬────────────────────────────┘
                         ↓
                    输出完成报告
```

## Phase 0: 准备

### 0.1 解析需求

```
输入：Jira需求ID或需求标题

示例输入：
  - CSS-1234
  - CSS-1234-用户登录功能
  - 【CSS-1234】用户登录功能

解析结果：
  - 需求ID: CSS-1234
  - 简短描述: 用户登录
```

### 0.2 确认开发模式

```
检测 features/ 目录是否已有同ID需求？

IF 已存在 THEN
  输出：
    "检测到已有需求 CSS-1234，选择开发模式："

    1. 增量开发 — 在现有基础上继续
    2. 全新功能 — 创建独立功能目录
    3. 重构 — 替换现有实现（需二次确认）

ELSE
  创建新目录 features/CSS-1234-用户登录/
END
```

### 0.3 创建目录和分支

```
目录结构：
features/
└── CSS-1234-用户登录/
    ├── PRD.md              （用户创建）
    ├── 需求理解确认.md      （Phase 2 输出）
    ├── 技术方案.md          （Phase 3 输出）
    ├── 任务拆解表.md        （Phase 4 输出）
    └── code-review.md      （tech-code 输出）

分支：
  git checkout -b feature/CSS-1234-用户登录
```

## Phase 1: 需求理解

### 1.1 用户创建 PRD

```
要求用户创建：
features/{id}/PRD.md

PRD模板位置：@configs/templates/prd-template.md（如有）
PRD参考：@docs/guides/prd-analysis-guide.md
```

### 1.2 AI 分析需求

读取 PRD.md，分析：

- 需求背景
- 核心功能点
- 目标用户
- 验收标准
- 非功能需求

### 1.3 逐项确认（One question at a time）

```
每次只问一个确认项，等待用户回复后再问下一个。

确认项1 — 需求背景
"我理解的需求背景是：【简述背景】
  请确认：
    A. 正确
    B. 基本正确，需要补充
    C. 错误，请说明"

确认项2 — 核心功能点
"我识别到的核心功能点是：
  1. 用户名密码登录
  2. 短信验证码登录
  3. 记住登录状态

  请补充或纠正："

...（继续直到所有关键项确认）
```

## Phase 2: 歧义检测

调用 `ambiguity-check.md`：

### 2.1 识别模糊点

检查 PRD 中的：
- 模糊描述词（"快速"、"友好"等）
- 边界条件是否明确
- 异常场景是否覆盖
- 数据量级是否清晰

### 2.2 生成澄清问题

```
输出待澄清问题列表：

| 序号 | 模糊点 | 影响 | 优先级 |
|------|--------|------|--------|
| 1 | "快速登录"未定义时间要求 | 无法验收 | 高 |
| 2 | 异常场景未说明 | 安全风险 | 高 |
| 3 | 最大用户数未明确 | 性能测试基准 | 中 |
```

### 2.3 需求理解确认

```
整合所有理解和澄清结果，输出：

# 需求理解确认

## 一、需求背景
【确认后的背景描述】

## 二、核心功能
【确认后的功能列表 + 每个功能的验收标准】

## 三、已澄清问题
【已解决的模糊点】

## 四、待澄清问题
【仍需人工确认的问题】

请确认是否可以通过技术方案设计。
```

## Phase 3: 技术方案

<HARD-GATE>
在用户明确批准技术方案之前，禁止：
- 调用任何编码命令
- 写任何业务代码
- 创建任何实现文件
- 调用 tech-code 或其他实现类技能

无论任务看起来多简单，不得例外。
</HARD-GATE>

调用 `tech-design-guide.md`：

### 3.1 技术方案模板

```
输出位置：features/{id}/技术方案.md
模板位置：@configs/templates/tech-design.md
```

### 3.2 方案评审检查

技术方案完成后，自检：

- [ ] 架构设计清晰
- [ ] 接口设计完整
- [ ] 数据库设计评审通过
- [ ] 依赖系统已识别
- [ ] 风险点已评估

### 3.3 关键确认

```
技术方案输出后暂停，等待确认。

★ 关键确认点：
  - 架构设计是否符合预期？
  - 接口设计是否满足需求？
  - 风险是否可以接受？

输入"确认"继续，或提出具体修改意见。
```

## Phase 4: 任务拆解

调用 `task-breakdown.md`：

### 4.1 分层任务结构

```
Scope（范围）
  ↓
Epic 1: 用户认证
  ↓
  Story 1.1: 用户名密码登录
  Story 1.2: 短信验证码登录
  Story 1.3: 登录状态管理
  ↓
Epic 2: 用户信息
  ↓
  Story 2.1: 查看个人信息
  Story 2.2: 修改密码
  ...
```

### 4.2 Story 验收标准

每个 Story 必须包含验收标准（Acceptance Criteria）：

```
Story 1.1: 用户名密码登录

验收标准：
  - [ ] 输入正确用户名密码，登录成功
  - [ ] 输入错误密码，提示"密码错误"
  - [ ] 输入不存在用户，提示"用户不存在"
  - [ ] 连续5次登录失败，账户锁定15分钟
  - [ ] 登录成功跳转至首页
```

### 4.3 任务拆解表示例

| Epic | Story | Task | 类型 | 估时 | 依赖 |
|------|-------|------|------|------|------|
| Epic 1 | Story 1.1 | T-001 数据库表设计 | 后端 | 0.5d | - |
| Epic 1 | Story 1.1 | T-002 User实体类开发 | 后端 | 0.5d | T-001 |
| Epic 1 | Story 1.1 | T-003 登录接口开发 | 后端 | 1d | T-002 |
| Epic 1 | Story 1.1 | T-004 登录页面前端 | 前端 | 0.5d | T-003 |
| ... | ... | ... | ... | ... | ... |

## 各阶段验证

详见 `verification.md`：

| Phase | 验证项 | 通过标准 |
|-------|--------|----------|
| Phase 0 | 目录/分支创建成功 | git status 确认 |
| Phase 1 | PRD 可读 | 能提取功能点 |
| Phase 2 | 无高优先级歧义 | 已澄清或已记录 |
| Phase 3 | 方案评审通过 | 自检清单全部通过 |
| Phase 4 | 任务可执行 | 每Task有验收标准 |
