```markdown
## 九、决策记录

<HARD-GATE>
用户确认的每个关键决策必须记录在此章节。
后续 /tech:code 阶段必须引用这些决策 ID。
禁止擅自改变用户已确认的决策。
</HARD-GATE>

| 决策ID | 决策内容 | 确认人 | 日期 | 备注 |
|--------|----------|--------|------|------|
| D-01 | 【例：选用 Spring Boot 3.x】 | 张三 | 2026-03-27 | - |
| D-02 | 【例：数据库选型 MySQL 8.0】 | 张三 | 2026-03-27 | - |

**决策引用格式**（在代码/审查中必须标注）：
```
// 决策依据：D-01
// 决策依据：D-02
```

**禁止事项**：
- 禁止在实现阶段改变用户已确认的决策（如需改变，必须重新确认）
- 禁止实现与决策记录不符的方案
- 禁止"优化"用户明确否决的方案
```

## 方案评审检查

### 自检清单

| 检查项 | 说明 | 状态 |
|--------|------|------|
| 架构设计 | 分层清晰，无循环依赖 | ☐ |
| 接口设计 | RESTful 规范，路径合理 | ☐ |
| 数据库设计 | 表结构合理，索引适当 | ☐ |
| 异常处理 | 有明确异常处理方案 | ☐ |
| 日志设计 | 关键操作有日志 | ☐ |
| 监控设计 | 有关键指标监控 | ☐ |
| 依赖系统 | 依赖方已确认 | ☐ |
| 风险评估 | 风险有缓解措施 | ☐ |

## 关键确认点（物理 HARD-GATE）

技术方案完成后，**禁止用文字提示替代**，必须调用 `ask_followup_question` 工具：

<TOOL-REQUIREMENT>
ask_followup_question({
  title: "技术方案确认 — {需求ID}",
  questions: [
    {
      id: "arch",
      question: "架构设计与接口方案是否符合预期？",
      options: ["✅ 通过", "❌ 需要修改"],
      multiSelect: false
    },
    {
      id: "db",
      question: "数据库表结构设计是否合理？",
      options: ["✅ 通过", "❌ 需要修改"],
      multiSelect: false
    },
    {
      id: "risk",
      question: "风险评估与上线方案是否可行？",
      options: ["✅ 通过", "❌ 需要修改"],
      multiSelect: false
    }
  ]
})

全部选择「✅ 通过」→ 继续执行决策锁定（update_memory）。
任意选择「❌ 需要修改」→ 用户说明修改意见，修改方案后重新触发本确认流程。
</TOOL-REQUIREMENT>
