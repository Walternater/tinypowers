# tech-design-guide.md

## 作用

本文档说明 `/tech:feature` 在技术方案阶段应覆盖哪些内容，以及什么情况下才算可以进入任务拆解。

## 技术方案至少应包含什么

一份可执行的技术方案，至少应覆盖：
- 需求背景和目标
- 功能范围
- 技术架构
- 数据设计
- 接口设计
- 核心流程
- 风险与上线考虑
- 决策记录

## 模板位置

技术方案模板见：
- `@configs/templates/tech-design.md`

默认输出位置：
- `features/{id}/技术方案.md`

## 方案阶段最重要的事

不是“把文档写长”，而是把关键决策写清楚。

尤其要明确：
- 为什么这样设计
- 有哪些替代方案被放弃
- 哪些内容在编码阶段不能再擅自改

## 方案自检

至少检查：

- [ ] 架构边界清晰
- [ ] 接口设计完整
- [ ] 数据模型合理
- [ ] 异常与风险已考虑
- [ ] 依赖系统已识别

## 硬门禁：方案确认

技术方案完成后，必须通过 `ask_followup_question` 做显式确认，不能用普通文本提示替代。

确认通过后，才能进入决策锁定与任务拆解。

## 决策锁定

方案确认后，应把关键决策写入持久化记忆，并同步记录到 `技术方案.md`。

典型决策类型：
- D-01 架构 / 框架选型
- D-02 数据结构 / 表结构
- D-03 对外接口契约
- D-04 中间件或依赖选型
- D-05 特殊安全方案

原则：
- 已确认决策在 `/tech:code` 阶段不能被擅自推翻
- 如需修改，应重新确认并更新记录

## 一个好的技术方案是什么样

读完方案后，团队应该能明确：

1. 要实现的边界是什么？
2. 关键接口和数据模型是什么？
3. 最大风险在哪里？
4. 哪些设计决策已经锁定？
5. 下一步如何拆成任务？

## Gotchas

- **技术方案不做用户确认就拆任务**：用 AI 自己的理解替代用户意图 → 返工：方案完成后必须通过 `ask_followup_question` 获得明确确认
