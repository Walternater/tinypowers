# verification.md

## tech-feature 验证清单

本文件用于校验 `/tech:feature` 各阶段是否已经达到“可以进入下一阶段”的标准。

它不是自动化脚本本身，而是校验标准说明。

## 验证目标

`/tech:feature` 结束时，至少应保证：
- Feature 目录和分支已创建
- `PRD.md` 已存在且可读
- 高优先级歧义已澄清或被显式记录
- `技术方案.md` 已形成并完成确认
- `任务拆解表.md` 已形成并具备可执行性

## Phase 0：准备

| 验证项 | 验证方法 | 通过标准 |
|--------|----------|----------|
| 目录创建 | `test -d features/{id}` | 目录存在 |
| 分支创建 | `git branch --list "feature/{id}*"` | 分支存在 |
| 基础结构 | `ls features/{id}/` | 至少可放置 `PRD.md` |

## Phase 1：需求理解

| 验证项 | 验证方法 | 通过标准 |
|--------|----------|----------|
| PRD 存在 | `test -f features/{id}/PRD.md` | 文件存在 |
| PRD 可读 | 读取文件内容 | 能提取背景、功能点、验收标准 |
| 理解结果可表述 | 人工检查 | 能输出清晰的需求理解摘要 |

## Phase 2：歧义检测

| 验证项 | 验证方法 | 通过标准 |
|--------|----------|----------|
| 模糊点识别 | 检查 PRD 与澄清结果 | 高优先级模糊点已识别 |
| 澄清问题形成 | 检查问题列表 | 问题有影响说明和优先级 |
| 阻塞项显式化 | 人工检查 | 未澄清项已明确记录 |

## Phase 3：技术方案

| 验证项 | 验证方法 | 通过标准 |
|--------|----------|----------|
| 方案存在 | `test -f features/{id}/技术方案.md` | 文件存在 |
| 架构清晰 | 读取内容 | 有结构说明、边界、关键流程 |
| 接口完整 | 检查接口设计 | 覆盖主要功能点 |
| 风险已记录 | 检查风险章节 | 至少列出关键风险和应对 |
| 已完成确认 | 检查评审记录 | 方案获得明确确认 |

## Phase 4：任务拆解

| 验证项 | 验证方法 | 通过标准 |
|--------|----------|----------|
| 拆解表存在 | `test -f features/{id}/任务拆解表.md` | 文件存在 |
| 功能覆盖 | 对照技术方案 | 主要功能点都有对应任务 |
| 验收标准存在 | 检查任务或 Story | 都有可验证标准 |
| 依赖清晰 | 检查依赖关系 | 无明显循环或悬空依赖 |
| 可以执行 | 人工检查 | 任务粒度和顺序合理 |

## 快速验收思路

可以用下面这组问题快速判断：

1. 现在是否已经能明确说出“要做什么”？
2. 现在是否已经明确说出“怎么做”？
3. 现在是否已经明确说出“先做什么、后做什么”？
4. 如果换一个会话，别人能不能接着继续做？

四个问题都能回答“是”，通常说明 `/tech:feature` 已经准备好进入 `/tech:code`。

## 失败处理

| 阶段 | 典型失败原因 | 处理方式 |
|------|-------------|----------|
| Phase 0 | 目录或分支未创建 | 修复初始化或创建逻辑 |
| Phase 1 | PRD 缺失或信息不足 | 等待补充 PRD |
| Phase 2 | 高优先级歧义未解决 | 继续澄清，不进入方案设计 |
| Phase 3 | 方案未确认或关键内容缺失 | 修改方案后重新确认 |
| Phase 4 | 拆解不完整或依赖混乱 | 补充任务并重新检查 |
