**结论**: 模板给了一个可用的骨架，但项目特色信息需要人工填充。

### 2.2 AI-KNOWLEDGE.md

**生成内容覆盖**: YAML frontmatter、代码位置索引（部分）、状态机图、PR 检查清单

### 2.3 docs/business-domain.md

**缺失的关键内容**:
- **任务小类详表缺失**: 手写版有 175+ 类型分表，生成版只写了"21 个硬编码枚举"
- **架构图缺失**: 任务创建入口图、状态流转图、FullUpTask 流程图都没有
- **Kafka Consumer 不完整**: 没有 Topic 名称、没有处理逻辑说明
- **bargain 模块完全缺失**: 手写版经过多次迭代才补入，生成版直接漏掉
- **代码示例缺失**: 7.1/7.2/7.3 节的完整 Java 代码示例
- **协同任务说明不足**: 缺少策略模式（CollaborateTaskSubBusinessFactory）的说明

**结论**: 这是差距最大的文件。模板对"业务逻辑深度"的生成能力不足。

### 2.4 docs/infrastructure.md

**生成内容覆盖**: Dubbo/Kafka/Redis/DB 的配置项（从 yml 直接提取）、ETCD 配置数量  
**缺失的关键内容**:
- **smart-common 组件库缺失**: 这是手写版最大的补充内容之一（11 个小节）
- **测试规范缺失**: 手写版有 JUnit/Mockito/测试模板/命名规范等详细内容
- **日志规范缺失**: 级别使用、打印规范、MDC 链路追踪、Kafka 消费日志
- **版本升级指南缺失**: smart-common 升级步骤、配置变更检查清单
- **安全规范缺失**: 敏感数据处理、接口权限控制、SQL 注入防护
- **监控告警缺失**: Zipkin、Sentinel、LogPolice 的详细配置
- **环境端口表缺失**: local/dev/stage/preview/online 的端口对照

**结论**: 扫描只能从 `application.yml` 提取配置值，但无法生成"规范类"内容（测试、日志、安全、升级）。这些内容高度依赖团队约定，需要预置模板或从 `guides/` 目录提取。

### 2.5 docs/operations.md

**生成内容覆盖**: 常用命令、外部服务接口列表、定时任务类列表  
**缺失的关键内容**:
- **三方交互深度缺失**: HTTP 路径、超时标准、核心接口方法签名
- **定时任务 cron 缺失**: 只有类名，没有 cron 表达式和调度说明
- **错误码体系缺失**: 1001xxx 结构、常见错误码、异常处理
- **FAQ 缺失**: 任务创建问题、Kafka 消费问题、性能问题、数据一致性问题
- **开发规范补充缺失**: Kafka 消费规范、分布式锁使用规范

**结论**: 扫描能找到"有什么"（类名、接口名），但无法推断"怎么用"（cron、超时、FAQ）。

## 3. 扫描能力的边界总结

### 扫描能做好的是：
1. ✅ 技术栈识别（pom.xml 依赖）
2. ✅ 模块结构提取（目录树）
3. ✅ 核心类定位（按文件大小排序）
4. ✅ 枚举值列表（硬编码枚举）
5. ✅ 中间件配置项（application.yml 提取）
6. ✅ 外部服务接口名（thirdpart/ 目录扫描）
7. ✅ Kafka 消费者类名与 Topic 映射
8. ✅ 测试文件数量统计

### 扫描做不好的/需要人工补充的是：
1. ❌ 业务量化指标（日均任务量、坐席数）
2. ❌ 动态配置说明（secode_type.json 的 175+ 配置型任务）
3. ❌ 隐藏模块识别（bargain 有独立包结构，但需要显式判断是否纳入）
4. ❌ 架构图和流程图
5. ❌ 代码示例
6. ❌ 团队规范（测试、日志、安全、代码审查）
7. ❌ FAQ 和故障排查
8. ❌ 接口的 HTTP 路径和超时标准
9. ❌ 定时任务的 cron 表达式

## 4. 设计改进建议

### 建议 A: 增强扫描逻辑的 5 个规则

| 规则 | 说明 |
|------|------|
| **模块边界规则** | 若发现与主包平级的独立包（如 `com.guazi.znkf.bargain`），必须显式询问用户是否纳入知识库 |
| **动态配置规则** | 若发现 `docs/data/*.json` 或 `TaskConfig` 类，提取并说明"配置型 vs 硬编码型"任务 |
| **定时任务规则** | 扫描 job 类后，尝试从类注释或方法中提取 `@Scheduled(cron = "...")` 或 ElasticJob 配置 |
| **HTTP 接口规则** | 扫描 `thirdpart/` 接口时，尝试匹配 `@FeignClient` 或 `RestTemplate` 调用处，提取 URL 和超时 |
| **规范文档规则** | 若项目已有 `docs/guides/` 或 `CLAUDE.md`，提取并引用到 infrastructure.md 中 |

### 建议 B: 模板应区分"骨架"和"扩展槽"

在每一章的模板中，明确标注：
```markdown
## 1. 中间件配置详解
<!-- AUTO: 以下内容由 AI 从 application.yml 自动生成 -->
...

<!-- MANUAL: 以下内容需人工补充 -->
### 1.5 监控与告警
（请补充 Zipkin、Sentinel、LogPolice 配置）
```

这样用户确认时，一眼就知道哪些是可信的、哪些必须补。

### 建议 C: 引入"团队规范预置包"

由于 `docs/infrastructure.md` 中有大量团队通用规范（测试、日志、安全），建议技能支持：
- 检测项目是否已有 `docs/guides/` 目录
- 若有，将 guides 内容摘要合并到 infrastructure.md
- 若无，从 skill 内置的"Java 团队规范模板"中预填充

### 建议 D: `/knowledge:check` 的优先级规则

基于本次验证，/knowledge:check 应按以下优先级提示更新：

| 优先级 | 检测信号 | 建议动作 |
|--------|----------|----------|
| P0 | 新增独立包结构（bargain） | 补充 business-domain.md |
| P0 | 新增 5+ 个枚举值 | 同步 business-domain.md |
| P1 | 新增 `@Attribute` 配置项 | 同步 infrastructure.md |
| P1 | 新增 `thirdpart/*.java` | 同步 operations.md |
| P1 | 新增 `@KafkaListener` | 同步 business-domain.md |
| P2 | `application.yml` 新增中间件配置 | 同步 infrastructure.md |

## 5. 结论

### 核心发现

1. **纯扫描生成的初稿约为手写版本的 18%**，这是一个合理的"骨架"水平，但远未达到可用。
2. **最大的差距不在"事实遗漏"，而在"深度和上下文"**: 
   - 扫描能告诉你"有 BaseTaskBusiness.java"
   - 但无法告诉你"它是系统的心脏，4758行，承担任务创建、分配、完成等核心逻辑"
3. **bargain 模块是本次验证中最典型的漏网之鱼**: 它有完整的包结构，但因为是"新兴子系统"，未被主模板识别。
4. **团队规范类内容（测试、日志、安全）几乎完全缺失**: 这类内容不来自代码扫描，而来自组织约定。

### 对 design.md 的反馈

当前 design.md 的扫描阶段设计基本合理，但需要在以下方面加强：
- 明确增加"模块边界识别"和"动态配置识别"
- 在模板中引入 `<!-- AUTO -->` / `<!-- MANUAL -->` 标注机制
- 为 infrastructure.md 增加"团队规范预置包"的 fallback 策略
- 降低对全自动生成内容的预期，强化"半自动确认"的价值

### 下一步建议

1. 先按本报告的改进建议更新 `design.md`
2. 然后选择一个小型项目（如 `smart-common`）做第二轮验证
3. 两轮验证收敛后，再进入 SKILL.md 的正式编写
